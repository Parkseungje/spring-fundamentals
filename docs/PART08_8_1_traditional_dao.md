# PART 8 — 객체 설계의 진화 → IoC/DI: 8.1 전통 DAO의 문제

> 이 문서는 커리큘럼 PART 8의 소단원 중 **8.1 전통 DAO의 문제**만 다룬다.
> PART 8은 "DAO 코드 한 줄이 어떻게 Spring의 IoC/DI까지 진화하는지"를 하나의 `UserDao` 리팩토링 여정으로
> 배운다. 그 출발점인 8.1은 "모든 책임을 떠안은 로우레벨 코드"의 고통을 직접 본다. (8.2부터 개선)

---

## 0. 들어가기 전에 — 핵심 용어
- **DAO(Data Access Object)**: DB 접근(저장·조회 등)을 담당하는 객체. 여기선 `UserDao`.
- **JDBC**: 자바에서 DB에 접속해 SQL을 실행하는 표준 API(`Connection`, `PreparedStatement`, `ResultSet` 등).
- **Connection(커넥션)**: DB와의 연결 통로. 만들고(`getConnection`) 다 쓰면 닫아야(`close`) 한다.
- **드라이버 로딩**: 어떤 DB(H2/MySQL...)와 통신할지 결정하는 드라이버를 메모리에 올리는 것(`Class.forName`).
- **관심사(concern)**: '하나의 목적을 가진 코드 묶음'. 예: '연결 만들기'와 'SQL 실행'은 서로 다른 관심사다.
- 관련 SOLID(PART 1): **SRP**(클래스가 바뀌는 이유는 하나) · **OCP**(확장엔 열고 수정엔 닫기) · **DIP**(구체가 아니라 추상에 의존).

한 줄 그림: **8.1의 UserDao는 한 메서드가 연결·SQL·해제·예외를 다 떠안아, DB가 바뀌면 모든 메서드를
고쳐야 하는 구조다.** 이 고통을 8.2~8.6에서 단계적으로 푼다.

---

## 1. 학습 내용 — "동작은 하지만 유지보수가 지옥인" 코드

### 한 메서드가 떠안은 6가지 책임
전통적인 JDBC DAO의 `add()`는 보통 이렇게 생겼다.
```java
public void add(User user) throws ... {
    Class.forName("org.h2.Driver");                       // ① 드라이버 로딩
    Connection c = DriverManager.getConnection(URL, ...); // ② 접속 정보(URL·계정)
    PreparedStatement ps = c.prepareStatement(            // ③ SQL
        "insert into users(id, name) values (?, ?)");
    ps.setString(1, user.getId());                        // ④ 파라미터 바인딩
    ps.executeUpdate();                                   // ⑤ 실행
    ps.close(); c.close();                                // ⑥ 자원 해제
}
```
한 메서드 안에 **① 드라이버 로딩·접속 ② SQL ③ 바인딩 ④ 실행 ⑤ 자원 해제 ⑥ 예외 처리**가 전부 뒤엉켜 있다.

### 무엇이 문제인가 (SOLID로 보면)
이 구조의 문제는 "코드가 길다"가 아니라 **"추적성과 변경 비용"**이다.

1. **SRP 위반 — 바뀌는 이유가 여러 개.** 한 메서드가 '연결을 어떻게 만드는지'와 '무슨 SQL을 실행하는지'
   라는 서로 다른 이유로 바뀐다. 둘 중 하나만 바뀌어도 이 메서드를 건드려야 한다.
2. **OCP/DIP 위반 — DB가 바뀌면 모든 메서드를 수정.** add, get, delete ... 모든 메서드에 '드라이버 로딩 +
   접속' 코드가 들어 있다. DB를 MySQL로 바꾸거나 접속 정보가 바뀌면 **모든 메서드를 다 고쳐야 한다**
   (변경 1번 = 수정 N곳). 또 구체 클래스(`DriverManager`)에 직접 의존한다(추상에 의존하지 않음 = DIP 위반).
3. **중복 — 같은 연결 코드가 메서드마다 복붙.** add와 get의 첫 두 줄(드라이버 로딩 + 접속)은 글자 그대로
   똑같다. 이 중복이 위 2번(수정 N곳)의 직접적 원인이다.

> ★ 핵심은 "DB가 바뀌면 몇 군데를 고쳐야 하나?"이다. 8.1은 'N군데(모든 메서드)'다. 이 숫자를 8.2에서
> '한 군데'로, 8.4에서 '0군데'로 줄여가는 것이 PART 8 전체의 이야기다.

## 2. 실습으로 확인하기

> - **가설**: 8.1 UserDao는 동작은 정상이지만, '드라이버 로딩 + 접속' 코드가 add/get에 중복되어 있어
>   DB 교체 시 여러 곳을 고쳐야 하는 구조다.

### 코드 (`com.study.part08_ioc.s01_traditional`)
- `UserDao` — add/get에 연결·SQL·해제가 모두 들어 있고, 연결 코드가 두 메서드에 중복된 '전통 DAO'.
- `Main` — add 후 get으로 동작을 확인하고, 구조적 문제를 짚는다.

### 실행
아래 명령은 **프로젝트 루트(`C:\develop\study\spring-fundamentals`)에서 실행**한다.
```bash
./gradlew runStage -Pmain=com.study.part08_ioc.s01_traditional.Main
```

### 실행 결과 — 가설과 실제 비교
```
[8.1] 저장 후 조회 결과 = User{id=u1, name=홍길동}
=> 동작은 한다. 그러나 add/get에 '드라이버 로딩+접속' 코드가 중복되어 있다.
```
- 저장→조회가 정상 동작한다(기능은 문제없음). ✅
- 그러나 `UserDao.java`를 열어보면 add와 get에 연결 코드가 복붙돼 있다 → DB 교체 시 두 메서드를 모두
  고쳐야 한다. "기능은 되는데 구조가 나쁜" 상태를 코드로 확인했다.

## 3. 자기 점검

- **Q. 8.1 UserDao가 SRP를 위반한다고 보는 이유는?**
  - 내 답: 한 메서드(add/get)가 '연결 만들기'와 'SQL 실행'이라는 서로 다른 이유로 바뀐다. 변경 이유가
    둘 이상이면 SRP 위반.

- **Q. DB 접속 정보가 바뀌면 8.1에서는 몇 곳을 고쳐야 하나? 왜?**
  - 내 답: 연결 코드가 모든 메서드에 복붙돼 있으므로 '모든 메서드'를 고쳐야 한다(변경 1번 = 수정 N곳).
    이 중복이 8.2에서 제거할 대상이다.

- **Q. (예고) 이 문제를 가장 먼저 어떻게 줄일 수 있을까?**
  - 내 답: 중복된 '연결 만들기' 코드를 한 메서드로 추출하면, DB 정보 변경 시 그 한 곳만 고치면 된다
    (8.2 관심사의 분리 1단계).
