## 0. 들어가기 전에 — 핵심 용어
- **핵심 관심사(core concern)**: 그 클래스가 진짜 해야 할 일(비즈니스 로직). 예: 주문 처리, 검증.
- **부가 관심사(cross-cutting concern, 횡단 관심사)**: 여러 곳에 공통으로 필요한 부가 기능. 예: 로깅·트랜잭션·보안·캐싱.
- **흩어진 관심사**: 그 부가 관심사가 여러 클래스·메서드에 반복해서 흩뿌려진 상태.
- **흩어짐(Scattering)**: 같은 부가 코드가 '여러 메서드에' 흩뿌려진 현상(가로축 문제).
- **엉킴(Tangling)**: 한 메서드 '안에서' 핵심 로직과 부가 코드가 뒤엉킨 현상(세로축 문제).
- **AOP(Aspect-Oriented Programming, 관점 지향 프로그래밍)**: 흩어진 부가 관심사를 한 곳으로 모아 핵심 로직과 분리하는 기법.
- **로그 추적기**: 메서드 호출 시작/종료·실행 시간·예외를 기록하는 부가 기능(이 단원의 예제).

한 줄 그림: **로깅 같은 부가 관심사를 넣으려면 모든 메서드에 같은 코드를 박아야 한다(흩어진 관심사).
이 '변하지 않는 부가 코드'를 '변하는 비즈니스 로직'에서 분리하는 것이 AOP의 목표다.**

## 1. 학습 내용 — 부가 코드가 모든 메서드에 흩어진다

### 핵심 관심사 vs 부가 관심사
- **핵심 관심사**: 비즈니스 로직(주문 처리, 검증 등). 메서드마다 '다르다'.
- **부가 관심사**: 로깅·트랜잭션·보안·캐싱. 여러 메서드에 '똑같이' 필요하다.

문제는 부가 관심사가 **여러 메서드에 반복 등장**한다는 것이다(횡단 관심사). OOP의 상속·합성만으로는
이 '흩어짐'을 깔끔히 없애기 어렵다.

### 로우레벨의 고통 — 모든 메서드에 try-catch + 로그
로그 추적기(시작/종료·시간·예외)를 넣으려면 메서드마다 이런 패턴을 박아야 한다.
```java
public void order(String item) {
    TraceStatus status = tracer.begin("order");   // 부가
    try {
        ... 비즈니스 로직 ...                       // 핵심(한두 줄)
        tracer.end(status);                        // 부가
    } catch (Exception e) {
        tracer.exception(status, e);               // 부가
        throw e;
    }
}
```
- 비즈니스 로직(핵심)은 한두 줄인데 그 주위를 **똑같은 로그 코드(부가)** 가 감싼다.
- 메서드가 100개면 이 6~7줄 패턴을 **100군데 복붙**해야 한다 = "배보다 배꼽". 로그 형식 하나만 바뀌어도 100군데 수정.

### 문제를 두 축으로 — 엉킴(Tangling)과 흩어짐(Scattering)
AOP 교과서는 이 고통을 두 가지 표준 용어로 정확히 짚는다.
- **흩어짐(Scattering)**: 같은 부가 코드(로그)가 **여러 메서드에 흩뿌려짐**. order에도, validate에도, … 100군데.
  → "한 번 고치려면 100군데를 고쳐야 한다"(가로축 문제).
- **엉킴(Tangling)**: 한 메서드 **안에서 핵심(비즈니스)과 부가(로그)가 뒤엉킴**. 비즈니스 한두 줄이 try-catch·
  로그에 파묻혀, 메서드만 봐선 "진짜 하는 일"이 안 보임(세로축 문제).
```
            order()        validate()      payment()
로그 코드     ▓ (엉킴)        ▓               ▓
비즈니스      ░              ░               ░
로그 코드     ▓              ▓               ▓
            └──────── 같은 ▓가 옆으로 반복 = 흩어짐 ────────┘
```

### 관심사가 겹치면 고통이 '곱'으로 커진다
실무의 한 메서드에는 로깅 하나만 있는 게 아니다. 보통 **로깅 + 트랜잭션 + 보안(권한) + 캐싱**이 동시에 박힌다.
```java
void placeOrder(String user, String item) {
    log.begin(...);                       // 부가 1) 로깅
    if (!hasPermission(user)) throw ...;  // 부가 2) 보안
    tx.begin();                           // 부가 3) 트랜잭션
    try {
        decreaseStock(item);              // ===== 진짜 비즈니스 (두 줄) =====
        createOrder(item);
        tx.commit();                      // 부가 3)
        log.end(...);                     // 부가 1)
    } catch (Exception e) { tx.rollback(); log.exception(...); throw e; }  // 부가 1)+3)
}
```
- 비즈니스는 **두 줄**인데 부가 코드가 메서드를 가득 채운다(엉킴 ↑). 그리고 이 엉킨 덩어리가 **모든 메서드에
  또 반복**된다(흩어짐 ↑). 관심사 종류가 N개로 늘수록 고통은 곱으로 악화된다.

### 왜 OOP(상속/합성)만으로는 안 되나
"그냥 공통 코드를 빼면 되지 않나?" 싶지만 기존 OOP 기법으론 깔끔히 안 된다.
- **상속으로 빼기**: 부모에 로그 흐름을 두려 해도, 자바는 다중 상속이 안 되고 관심사(로깅·보안·트랜잭션)마다
  부모를 늘릴 수 없다. 게다가 "흐름 사이에 비즈니스를 끼우는" 구조라 부모에 종속된다.
- **합성/위임으로 빼기**: 로그 로직을 헬퍼로 빼도, 결국 **각 메서드에서 begin/end 호출 코드를 또 박아야** 해서
  흩어짐이 그대로 남는다.
- 디자인 패턴(템플릿 메서드/전략/콜백)으로 개선해도 **결국 원본 메서드를 수정**해야 한다(12.3에서 직접 확인).
→ 그래서 "부가 관심사를 한 곳(Aspect)에 모아, 원본을 안 건드리고 필요한 메서드에 **자동으로 끼워 넣는**"
  새 방법이 필요하다. 그게 AOP다. 이후 12.3(패턴의 한계) → 12.4(프록시) → 12.9(@Aspect)로 단계적으로 구현한다.

> ★ AOP는 OOP를 '대체'하지 않는다. 핵심 로직은 여전히 OOP로 잘 설계하고, 거기에 흩어지는 부가 관심사만
> AOP로 분리해 '보완'하는 것이다(남용하면 흐름 추적이 어려워짐 — 12.9).

## 2. 실습으로 확인하기

> - **가설 1**: 로그 추적기는 잘 동작하지만, 그 코드(begin/end/exception+try-catch)가 모든 메서드에 흩어져
>   박혀 비즈니스 로직보다 길어진다(흩어짐).
> - **가설 2**: 한 메서드에 로깅+보안+트랜잭션이 겹치면 비즈니스 두 줄이 부가 코드에 파묻힌다(엉킴).

### 코드 (`com.study.part12_aop.s01_aop_motivation`)
- `LogTracer` — begin/end/exception(부가 기능).
- `OrderService` — order/validate 두 메서드가 각각 같은 로그 보일러플레이트로 감싸여 있음(흩어짐).
- `Main` — 정상/예외 주문 실행(흩어짐 관찰).
- `TangledOrderService` / `Example2_TangledConcerns` — 한 메서드에 로깅+보안+트랜잭션이 엉킨 모습(엉킴 관찰).

### 실행
**프로젝트 루트(`C:\develop\study\spring-fundamentals`)에서 실행**한다.
```bash
./gradlew runStage -Pmain=com.study.part12_aop.s01_aop_motivation.Main
./gradlew runStage -Pmain=com.study.part12_aop.s01_aop_motivation.Example2_TangledConcerns
```

### 실행 결과 — 가설과 실제 비교
```
== 정상 주문 ==
--> [OrderService.order] 시작
--> [OrderService.validate] 시작
    [비즈니스] 노트북 검증 통과
<-- [OrderService.validate] 종료 (1ms)
    [비즈니스] 노트북 주문 처리
<-- [OrderService.order] 종료 (9ms)
== 검증 실패(예외) 주문 ==  -> <X- 예외 로그까지 정상
```
- 로그(시작/종료/예외/시간)는 잘 찍힌다. ✅ 하지만 `OrderService`를 보면 order/validate **둘 다** 똑같은
  로그 보일러플레이트로 감싸여 있다 → 메서드마다 반복(흩어짐). 이 흩어짐을 없애는 게 AOP의 동기.

예제2 (엉킴) 실측:
```
== 정상(admin) ==
--> [TangledOrderService.placeOrder] 시작     <- 로깅
    [트랜잭션] begin                          <- 트랜잭션
    [비즈니스] 노트북 재고 차감                 <- 핵심(이 두 줄뿐)
    [비즈니스] 노트북 주문 생성
    [트랜잭션] commit
<-- [TangledOrderService.placeOrder] 종료
== 권한 없음(guest) ==  -> 보안에서 막혀 비즈니스 도달 못 함
```
- 비즈니스 두 줄을 로깅+보안+트랜잭션이 둘러싸 메서드를 가득 채운다(엉킴). 이 덩어리가 모든 메서드에 또
  반복(흩어짐)된다. ✅ → 관심사가 겹칠수록 고통이 곱으로 커진다 = AOP의 더 강한 동기.

## 3. 자기 점검

- **Q. 핵심 관심사와 부가(횡단) 관심사의 차이는?**
  - 내 답: 핵심은 그 메서드가 진짜 할 일(비즈니스, 메서드마다 다름), 부가는 로깅·트랜잭션·보안처럼 여러
    메서드에 공통으로 필요한 기능(흩어짐).

- **Q. 로그 추적기를 OOP로만 넣을 때의 고통은?**
  - 내 답: 모든 메서드에 begin/end/exception+try-catch를 복붙해야 하고, 형식 하나 바꾸면 그 모든 곳을
    수정해야 한다(100개 메서드=100군데). 비즈니스보다 부가 코드가 더 길어진다.

- **Q. 엉킴(Tangling)과 흩어짐(Scattering)의 차이는?**
  - 내 답: 엉킴은 한 메서드 '안'에서 핵심과 부가가 뒤엉킨 것(세로축), 흩어짐은 같은 부가 코드가 '여러
    메서드에' 반복된 것(가로축). AOP는 이 둘을 동시에 푼다(부가를 한 곳에 모아 자동으로 끼움).

- **Q. 부가 관심사가 여러 개 겹치면 왜 더 심각한가?**
  - 내 답: 한 메서드에 로깅+보안+트랜잭션+캐싱이 동시에 박히면 비즈니스 몇 줄이 부가 코드에 파묻히고(엉킴 ↑),
    그 덩어리가 모든 메서드에 반복된다(흩어짐 ↑). 관심사 수만큼 고통이 곱으로 커진다.

- **Q. 왜 OOP(상속/합성)만으로는 부가 관심사를 못 떼나?**
  - 내 답: 상속은 다중 상속 불가·관심사마다 부모를 못 늘림, 합성/위임도 결국 각 메서드에서 호출 코드를 또
    박아야 해 흩어짐이 남는다. 디자인 패턴도 원본 수정이 필요(12.3). 그래서 원본을 안 건드리고 자동으로
    끼워 넣는 AOP가 필요하다. (AOP는 OOP를 대체가 아니라 보완.)

- **Q. AOP가 풀려는 문제를 한 문장으로?**
  - 내 답: 여러 메서드에 흩어지고(scattering) 엉킨(tangling) 부가 관심사(변하지 않는 코드)를 한 곳에 모아,
    핵심 로직(변하는 코드)에서 분리하고 필요한 곳에 자동으로 끼워 넣는 것.
