# PART 12 — 프록시의 진화와 Spring AOP: 12.4 프록시 개념과 조건

> 이 문서는 커리큘럼 PART 12의 소단원 중 **12.4 프록시 개념과 조건**을 다룬다.
> 12.3에서 세 패턴(템플릿 메서드/전략/템플릿 콜백)이 끝내 못 푼 '원본 코드 수정' 한계를, 프록시가 어떻게
> 'RealOrderService를 한 줄도 안 건드리고' 넘는지 본다. 프록시의 두 기능(부가 기능/접근 제어)도 구분한다.

---

## 0. 들어가기 전에 — 핵심 용어
- **프록시(Proxy, 대리인)**: 클라이언트와 진짜 객체 사이에 끼는 중간 객체. 클라이언트는 프록시를 호출하고, 프록시가 부가 처리 후 진짜 객체에 위임한다.
- **진짜 객체(real subject / target)**: 실제 비즈니스 로직을 가진 원본 객체. 여기선 `RealOrderService`.
- **위임(delegate)**: 프록시가 받은 호출을 진짜 객체에게 그대로 넘겨 실제 일을 시키는 것.
- **DIP(의존 역전 원칙)**: 구체 클래스가 아니라 인터페이스(추상)에 의존하는 설계(PART 8.4).
- **DI(의존성 주입)**: 객체가 쓸 의존을 밖에서 넣어 주는 것. 여기선 생성자로 주입.
- **부가 기능 / 접근 제어**: 프록시의 두 용도. 부가 기능=로그·시간 측정, 접근 제어=캐싱·권한·지연 로딩.

한 줄 그림: **프록시는 '진짜 객체와 같은 인터페이스'를 구현해 클라이언트를 속이고, 호출을 가로채 부가
처리(로그) 후 위임하거나 접근을 제어(캐싱)한다. 클라이언트가 인터페이스에만 의존하고 DI로 주입받으면,
원본·클라이언트 코드를 0줄 바꾸고도 기능을 끼울 수 있다 — 이것이 12.3 '원본 수정' 한계의 해법이다.**

## 1. 학습 내용

### 1-1. 프록시란 — 클라이언트와 진짜 객체 사이의 '대리인'
프록시는 일상어로 '대리인'이다. 클라이언트는 진짜 객체에 직접 가지 않고 대리인(프록시)에게 부탁한다.
대리인은 필요한 부가 처리를 한 뒤 진짜 객체에 일을 넘긴다(위임).
```
[직접 호출]   클라이언트 ───────────────→ 진짜 객체
[프록시 경유] 클라이언트 ──→ 프록시 ──위임──→ 진짜 객체
                            (여기서 로그/캐싱/권한 등 부가 처리)
```
비유: 연예인(진짜 객체)에게 섭외하려면 매니저(프록시)를 거친다. 매니저가 일정 확인·계약(부가 처리)을
하고 연예인에게 전달한다. 의뢰인(클라이언트)은 매니저와 연예인을 같은 창구로 느낀다.

### 1-2. 성립 조건 — "클라이언트가 진짜인지 프록시인지 몰라야 한다"
프록시가 성립하려면 클라이언트가 **진짜 객체와 프록시를 구분하지 못해야** 한다. 그래야 둘을 바꿔 끼워도
클라이언트가 안 바뀐다. 그 조건은 두 가지다.
- 조건 1 — **진짜 객체와 프록시가 '같은 인터페이스'를 구현**한다(`OrderService`). 그래야 같은 타입으로 취급된다.
- 조건 2 — **클라이언트는 인터페이스에만 의존하고, 구현체는 DI로 주입**받는다. 클라이언트는 무엇이 들어올지 모른다.
```java
public class Client {
    private final OrderService orderService;       // 인터페이스에 의존(DIP)
    public Client(OrderService orderService) {     // 무엇이 올지 모른 채 주입(DI)
        this.orderService = orderService;
    }
}
// 주입만 바꾼다 -> 클라이언트 코드는 0줄 변경
new Client(new RealOrderService());        // 진짜
new Client(new LogProxy(real));            // 부가 기능 프록시
new Client(new CachingProxy(real));        // 접근 제어 프록시
```
> ★ 왜 인터페이스 + DI가 필수인가 — 클라이언트가 `new RealOrderService()`처럼 구체 클래스를 직접 만들면,
> 프록시로 바꾸려면 클라이언트 코드를 고쳐야 한다(원본 수정 부활). 인터페이스 의존 + 외부 주입이라야
> "주입만 갈아끼우면 끝"이 된다. (PART 8.4~8.6에서 DAO에 ConnectionMaker를 주입한 그 구조와 똑같다.)

### 1-3. 프록시의 두 기능 — 부가 기능 vs 접근 제어
프록시가 호출을 가로챈 뒤 할 수 있는 일은 크게 두 가지다.
- **부가 기능(LogProxy)**: 진짜 호출 앞뒤에 로그·시간 측정 등을 더한다. **호출은 항상 진짜 객체로 통과**시킨다.
  ```
  begin 로그 -> target.order() 위임 -> end 로그   (진짜 호출은 그대로 일어남)
  ```
- **접근 제어(CachingProxy)**: 진짜 객체로의 **접근 여부 자체를 결정**한다. 캐시에 값이 있으면 **진짜 객체를
  아예 안 부른다**. 권한 프록시라면 권한 없을 때 진짜 호출을 막고 예외를 던진다. 지연 로딩이면 처음 쓸 때만 만든다.
  ```
  캐시 있음? -> 예: 캐시 즉시 반환(진짜 호출 생략!) / 아니오: target.findStock() 후 캐시에 저장
  ```
- 결정적 차이: 부가 기능은 "항상 위임 + 꾸미기", 접근 제어는 "위임할지 말지를 통제". 모양(같은 인터페이스
  구현 + target 보유)은 같고 **의도가 다르다** → 이 의도 차이가 12.5(프록시 패턴 vs 데코레이터 패턴)로 이어진다.

### 1-4. 12.3 한계를 어떻게 넘는가 / 남는 한계 → 12.6
- 12.3 세 패턴은 부가 기능을 붙이려면 호출부를 `execute(...)`로 감싸 **원본을 수정**해야 했다.
- 프록시는 진짜 객체(`RealOrderService`)도 클라이언트(`Client`)도 **한 줄도 수정하지 않는다**. 오직 main에서
  주입 대상만 바꾼다. → '원본 수정' 한계를 넘었다.
- **남는 한계(12.6의 동기)**: 지금은 프록시를 **손으로** 만들었다(`LogProxy`가 `OrderService`의 모든 메서드를
  일일이 구현). 인터페이스 메서드가 많거나 프록시가 필요한 클래스가 많으면 **프록시 클래스가 폭발**한다.
  이 반복을 자동화하는 게 '동적 프록시'(Reflection → JDK 동적 프록시 → CGLIB, 12.6)다.

## 2. 실습으로 확인하기

> - **가설**: 진짜 객체를 ①그대로 ②LogProxy로 ③CachingProxy로 감싸 주입하면, Client 코드는 동일한데
>   동작이 달라진다(로그 추가 / 캐싱). 부가 기능 프록시는 항상 위임하고, 접근 제어 프록시는 두 번째 조회에서 진짜 호출을 생략한다.

### 코드 (`com.study.part12_aop.s04_proxy`)
- `OrderService`(인터페이스), `RealOrderService`(진짜, 순수 비즈니스), `Client`(인터페이스 의존+DI).
- `LogProxy`(부가 기능), `CachingProxy`(접근 제어), `Trace`(로그 헬퍼).
- `Example1_NoProxy` / `Example2_LogProxy` / `Example3_CachingProxy` — 주입 대상만 바꿔 비교.

### 실행
**프로젝트 루트(`C:\develop\study\spring-fundamentals`)에서 실행**한다.
```bash
./gradlew runStage -Pmain=com.study.part12_aop.s04_proxy.Example1_NoProxy
./gradlew runStage -Pmain=com.study.part12_aop.s04_proxy.Example2_LogProxy
./gradlew runStage -Pmain=com.study.part12_aop.s04_proxy.Example3_CachingProxy
```

### 실행 결과 — 가설과 실제 비교 (실측)
예제1 (기준선): 로그 없음, 재고 조회 두 번 다 실제 수행.
```
    [비즈니스] 노트북 재고를 '실제로' 조회(느림)...
    [비즈니스] 노트북 재고를 '실제로' 조회(느림)...   <- 두 번 다 진짜 조회
```
예제2 (부가 기능): 호출 앞뒤로 로그, 단 진짜 조회는 여전히 두 번(항상 위임).
```
--> [OrderService.findStock] 시작
    [비즈니스] 노트북 재고를 '실제로' 조회(느림)...
<-- [OrderService.findStock] 종료 (205ms)
--> [OrderService.findStock] 시작                    <- 두 번째도 진짜 호출(205ms급)
```
예제3 (접근 제어): 첫 조회만 실제, 두 번째는 캐시 즉시 반환(진짜 호출 생략).
```
    [비즈니스] 노트북 재고를 '실제로' 조회(느림)...   <- 첫 번째만 진짜
    [캐시] 노트북 재고를 캐시에서 즉시 반환(진짜 조회 생략)   <- 두 번째는 캐시
```
- 세 예제 모두 **`Client`와 `RealOrderService` 코드는 동일**하고, main의 주입 대상만 바뀌었다. ✅ → 인터페이스
  + DI 위에서 프록시를 끼우면 원본·클라이언트 수정 없이 기능이 바뀐다. 부가 기능(항상 위임)과 접근 제어
  (호출 통제)의 차이도 두 번째 조회 동작으로 확인된다.

## 3. 자기 점검

- **Q. 프록시란 무엇이고, 성립 조건 두 가지는?**
  - 내 답: 클라이언트와 진짜 객체 사이의 대리인. 조건은 ①진짜 객체와 프록시가 같은 인터페이스를 구현,
    ②클라이언트가 인터페이스에만 의존하고 DI로 주입받음. 그래야 클라이언트가 둘을 구분 못 해 코드 변경 0.

- **Q. 프록시는 12.3 패턴들이 못 푼 무엇을 푸나?**
  - 내 답: '원본 코드 수정'. 12.3은 호출부를 execute로 감싸야 했지만, 프록시는 RealOrderService도 Client도
    안 건드리고 주입만 바꿔 기능을 더한다.

- **Q. 부가 기능 프록시와 접근 제어 프록시의 차이는?**
  - 내 답: 부가 기능(LogProxy)은 항상 진짜 객체에 위임하고 앞뒤로 로그 등을 더한다. 접근 제어(CachingProxy)는
    진짜 객체로의 접근 여부 자체를 통제한다(캐시 있으면 진짜 호출 생략, 권한 없으면 차단). 모양은 같고 의도가 다르다.

- **Q. 직접 만든 프록시의 한계와, 그것이 무엇으로 이어지나?**
  - 내 답: 인터페이스 메서드마다 손으로 위임 코드를 써야 해 프록시 클래스가 폭발한다. 이 반복을 자동화하는
    것이 동적 프록시(Reflection/JDK 동적 프록시/CGLIB, 12.6)다.
