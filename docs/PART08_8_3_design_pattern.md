# PART 8 — 객체 설계의 진화 → IoC/DI: 8.3 디자인 패턴의 적용과 한계

> 이 문서는 커리큘럼 PART 8의 소단원 중 **8.3 디자인 패턴의 적용과 한계**를 다룬다.
> 8.2에서 만든 추상클래스 구조가 사실 두 고전 디자인 패턴의 동시 적용임을 '인식'하고, 그 상속 기반
> 방식의 한계를 정리한다. (8.2는 `PART08_8_2_separation.md` 참고)

---

## 0. 들어가기 전에 — 핵심 용어
- **디자인 패턴(design pattern)**: 자주 나오는 설계 문제에 대한 검증된 해법(이름 붙은 구조).
- **템플릿 메소드 패턴**: 슈퍼클래스가 '흐름(뼈대)'을 고정하고, 변하는 한 부분만 서브클래스가 구현.
- **팩토리 메소드 패턴**: '객체 생성'을 서브클래스에 위임하는 패턴(무엇을 만들지는 자식이 결정).
- **컴파일 타임 결합(compile-time coupling)**: "어떤 구현을 쓸지"가 소스 코드(`new X()`)에 박혀, 바꾸려면 코드 수정·재컴파일이 필요한 상태.
- **단일 상속(single inheritance)**: 자바 클래스는 부모를 하나만 가질 수 있다는 제약.
- **강결합(tight coupling)**: 한쪽이 바뀌면 다른 쪽도 영향을 받는, 단단히 묶인 관계.

한 줄 그림: **8.2의 (추상클래스 + 자식 구현) 구조 = 템플릿 메소드 + 팩토리 메소드 패턴. 잘 동작하지만
'상속'이라 ①단일 상속 ②컴파일 타임 결합 ③부모-자식 강결합이라는 한계가 있다 → 8.4에서 합성으로 푼다.**

---

## 1. 학습 내용 — 패턴으로 보기, 그리고 한계

### 8.2 구조 = 두 패턴의 동시 적용
8.2에서 만든 `AbstractUserDao`(흐름) + `NUserDao`/`DUserDao`(연결 구현) 구조는 새로 발명한 게 아니라,
이미 이름 붙은 두 패턴이 겹쳐 있는 것이다.

- **템플릿 메소드 패턴**: 부모 `AbstractUserDao`가 add/get의 '흐름(SQL 실행 절차)'을 고정하고, 변하는
  한 부분(`getConnection`)만 자식이 구현한다. → Spring **JdbcTemplate**의 'Template'이 이 패턴 이름에서 왔다.
- **팩토리 메소드 패턴**: 객체 생성(Connection 만들기)을 자식의 `getConnection`에 위임한다(무엇을·어떻게
  만들지는 자식이 결정). → Spring **BeanFactory**의 'Factory'가 이 개념이다.

> 즉 "DAO 리팩토링" 하나로 GoF의 두 패턴을 동시에 경험한 셈이고, 그 이름들이 Spring 핵심 클래스
> (JdbcTemplate·BeanFactory)의 어원이라는 점이 8.3의 묘미다.

### 상속 기반 분리의 한계 3가지
패턴은 적절했지만 '상속'으로 구현했다는 점에서 한계가 따라온다.

1. **단일 상속 제약 (한계①)**: 자바는 부모를 하나만 가질 수 있다. UserDao가 다른 클래스(공통 기능 등)를
   상속해야 하면, AbstractUserDao와 둘 다는 불가능하다. '연결 분리'를 위해 유일한 상속 카드를 써버린 것.
   (코드: `SingleInheritanceLimitation` — BaseComponent를 상속한 클래스는 AbstractUserDao를 또 못 상속.)
2. **컴파일 타임 결합 (한계②)**: "어떤 연결을 쓸지"가 `new NUserDao()`처럼 소스에 박힌다. N→D로 바꾸려면
   `new`를 고치고 다시 컴파일해야 한다. "실행 시점에 갈아끼우기"가 안 된다.
3. **부모-자식 강결합 (한계③)**: 부모 `AbstractUserDao`가 바뀌면 모든 자식이 영향을 받는다. 흐름과
   연결이 '상속'으로 단단히 묶여 있다.

→ 이 세 한계의 공통 원인은 '상속'이다. 8.4에서 **상속(extends) 대신 인터페이스 구현 + 합성(필드로 들고
위임)** 으로 바꾸면, 인터페이스는 여러 개 구현 가능(①해결)·구현체를 외부에서 주입(②③해결)할 수 있다.

## 2. 실습으로 확인하기

> - **가설 1**: 8.2 구조를 실행하면 '템플릿 메소드(부모 흐름) + 팩토리 메소드(자식 생성)'가 함께 동작한다.
> - **가설 2**: N→D 교체에 `new`를 고쳐야 한다(컴파일 타임 결합). 다른 클래스 상속이 필요하면 막힌다(단일 상속).

### 코드 (`com.study.part08_ioc.s03_design_pattern`)
- `Main` — 8.2의 `AbstractUserDao`/`NUserDao`/`DUserDao`를 그대로 가져와 패턴을 해설하고 한계②를 시연.
- `SingleInheritanceLimitation` — 단일 상속(한계①)을 코드로 드러냄(다른 클래스를 상속하면 AbstractUserDao 상속 불가).

### 실행
**프로젝트 루트(`C:\develop\study\spring-fundamentals`)에서 실행**한다.
```bash
./gradlew runStage -Pmain=com.study.part08_ioc.s03_design_pattern.Main
./gradlew runStage -Pmain=com.study.part08_ioc.s03_design_pattern.SingleInheritanceLimitation
```

### 실행 결과 — 가설과 실제 비교
```
[8.3] 8.2 구조를 패턴으로 분석 + 한계
  실행 결과 = User{id=p1, name=패턴분석}
  -> add/get 흐름(부모) = 템플릿 메소드, getConnection(자식) = 팩토리 메소드.
  [한계②] N->D 교체에 'new'를 바꿔야 했다(코드 수정+재컴파일 필요): User{id=p2, name=교체하려면코드수정}
```
```
[8.3 한계①] 단일 상속 제약
  MyComponent는 BaseComponent를 상속함 -> AbstractUserDao를 '또' 상속 불가(자바 단일 상속).
```
- 가설 1: 부모 흐름 + 자식 생성이 함께 동작(두 패턴 확인). ✅
- 가설 2: 교체에 코드 수정 필요(②), 다른 상속과 충돌(①). ✅ → 8.4에서 해결.

## 3. 자기 점검

- **Q. 8.2 구조에 적용된 두 디자인 패턴과, 각 패턴이 Spring 어느 클래스 이름으로 이어지나?**
  - 내 답: 템플릿 메소드 패턴(부모 흐름+자식 구현) → JdbcTemplate. 팩토리 메소드 패턴(객체 생성을 자식에 위임) → BeanFactory.

- **Q. 상속 기반 분리의 한계 3가지는?**
  - 내 답: ①단일 상속 제약 ②컴파일 타임 결합(교체에 코드 수정) ③부모-자식 강결합(부모 변경이 자식에 파급).

- **Q. 이 한계들의 공통 원인은 무엇이고, 8.4는 어떻게 푸는가?**
  - 내 답: 공통 원인은 '상속'. 8.4는 상속 대신 인터페이스 구현 + 합성으로 바꿔, 여러 인터페이스 구현
    가능(①)·구현체를 외부에서 런타임 주입(②③)으로 해결한다.
