# PART 11 — ORM/JPA와 트랜잭션 추상화: 11.2 ORM 패러다임 (객체-관계 미스매치)

> 이 문서는 커리큘럼 PART 11의 소단원 중 **11.2 ORM 패러다임**을 다룬다.
> 객체(OOP)와 관계형 DB는 서로 다른 패러다임이라 '미스매치'가 있다. 그 간극을 손으로 메우는 고통을 직접
> 겪고, 그것을 자동으로 메워주는 ORM의 필요성을 이해한다. (JPA는 11.3~)

---

## 0. 들어가기 전에 — 핵심 용어
- **패러다임(paradigm)**: 세상을 모델링하는 큰 사고방식. 여기선 '객체지향' vs '관계형'.
- **객체-관계 미스매치(impedance mismatch)**: 객체 모델과 관계형 모델의 구조적 차이(아래 표).
- **ORM(Object-Relational Mapping)**: 객체는 객체대로, DB는 DB대로 설계하고 **프레임워크가 중간에서 매핑**해 주는 기술. (자바: Hibernate, EclipseLink)
- **객체 그래프(object graph)**: 객체가 참조로 이어진 망(예: order → member). DB에선 외래 키로 표현.
- **영속성 컨텍스트(persistence context)**: ORM이 "같은 PK는 같은 객체"로 관리하는 1차 캐시(11.3~에서 본격).

한 줄 그림: **객체(참조·==·상속)와 DB(외래 키·PK·테이블)는 구조가 달라, 그 간극을 메우는 코드를 매번
손으로 써야 한다. ORM은 객체 그래프 조립과 동일성 관리를 자동화해 그 고통을 없앤다.**

## 1. 학습 내용 — 객체와 DB는 어긋난다

### 미스매치 여섯 측면
| 측면 | 객체(OOP) | 관계형 DB |
|---|---|---|
| 모델링 | 상태 + 행동(메서드) | 행과 열(데이터만) |
| 상속 | 있음 | 없음(SINGLE_TABLE/JOINED/TABLE_PER_CLASS로 흉내) |
| 연관 | 참조(`order.member`) | 외래 키(`member_id` 숫자) |
| 식별 | `==`(인스턴스 동일성) | PK(값 동일성) |
| 탐색 | 참조를 타고 **자유롭게** 이동(`order.member.team...`) | **JOIN한 범위**까지만 |
| 세분성 | 값 객체로 **잘게** 쪼갬(`Address`) | 보통 컬럼 몇 개로 펼침 |

### 그래서 생기는 '손으로 메우는 고통'
- **연관 미스매치**: DB는 `member_id`(숫자)만 준다. 이걸 객체 그래프(`order.member`)로 만들려면 회원을
  따로 조회해 `new Member(...)` 하고 `order.setMember(member)`로 **손수 연결**해야 한다(객체 그래프 수동 조립).
- **식별 미스매치**: 같은 회원(PK=1)을 두 번 조회하면 DB상 같은 행이지만, 자바에선 `new`를 두 번 해서
  **서로 다른 객체(`==` false)** 가 된다. "같은 데이터인데 다른 객체"라 동일성 관리가 꼬인다.
- **★ 탐색 미스매치**: 객체에선 `order.getMember().getTeam().getLeader()`처럼 참조를 **자유롭게 타고** 들어가는
  게 자연스럽다. 그런데 SQL Mapper에선 **"처음 실행한 그 SQL이 JOIN/조회한 만큼만"** 객체가 채워진다. 회원을
  조회하지 않은 `Order`는 `member`가 **null**이라, `order.getMember().getName()`처럼 더 타고 가면 NullPointerException이 난다.
  → "이 객체 그래프를 어디까지 믿고 탐색해도 되나?"를 알 수 없다(SQL마다 채워진 범위가 다름). 그래서 객체 그래프를
  마음껏 신뢰하지 못한다. 이 한계가 JPA **지연 로딩(Lazy Loading)** 과 "신뢰할 수 있는 객체 그래프"의 직접적 동기다(11.3~).
- **세분성(granularity) 미스매치**: 객체는 `Member`가 `Address`(도시·우편번호) 같은 **값 객체**를 품어 더 잘게
  표현할 수 있다. 반면 DB는 보통 `city`, `zipcode` 컬럼으로 **펼쳐** 둔다(표현 단위가 다름). 그래서 객체 ↔ 테이블
  사이에서 "객체 한 덩어리 ↔ 컬럼 여러 개"를 손으로 변환해야 한다. (JPA는 `@Embedded` 값 타입으로 이를 매핑한다.)

이런 변환 코드를 쿼리마다 반복하는 게 SQL Mapper(JdbcTemplate) 시대의 한계였다.

### ORM이 메우는 것 (그리고 단점)
**ORM**은 객체는 객체대로, DB는 DB대로 두고 프레임워크가 중간에서 매핑한다.
- 객체 그래프를 **자동 조립**(연관을 따라가면 알아서 조회·연결, Lazy Loading — 11.3).
- **같은 PK는 같은 객체**로 관리(영속성 컨텍스트) → 식별 미스매치 해소.
- SQL을 완전히 대체하진 않는다(복잡한 쿼리는 여전히 SQL/Querydsl 필요).
- 단점: **학습곡선**, **N+1 문제**(연관을 무심코 조회하면 쿼리 폭증), **튜닝 어려움**(생성 SQL이 숨겨짐).

## 2. 실습으로 확인하기

> - **가설**: raw JDBC로는 (A) 외래 키를 객체 참조로 '손수' 조립해야 하고, (B) 같은 PK를 두 번 조회하면
>   서로 다른 객체(`==` false)가 되며, (C) 조회 안 한 연관은 null이라 객체 그래프를 더 탐색할 수 없다.

### 코드 (`com.study.part11_jpa.s02_orm_mismatch`)
- 도메인: `Member`, `Order`(member를 '참조'로 보유).
- `Main` — (A) order를 읽어 member를 따로 조회·연결(객체 그래프 수동 조립), (B) 같은 PK 회원 두 번 조회 후
  `==` 비교, (C) item만 읽어 member가 null인 '반쪽 객체'로 탐색 한계 확인.

### 실행
**프로젝트 루트(`C:\develop\study\spring-fundamentals`)에서 실행**한다.
```bash
./gradlew runStage -Pmain=com.study.part11_jpa.s02_orm_mismatch.Main
```

### 실행 결과 — 가설과 실제 비교
```
(A) 조립된 객체 그래프 = Order{id=1, item=노트북, member=Member{id=1, name=홍길동}}
    order.member.name = 홍길동  <- 회원을 따로 new 해서 setMember 해야 했다
(B) m1 == m2 ? false   <- 같은 PK(1)인데 'new'를 두 번 해 다른 인스턴스(객체 식별 != PK)
(C) item만 읽은 Order = Order{id=1, item=노트북, member=null}
    itemOnly.getMember() = null   <- 이 SQL은 회원을 안 채움 -> 더 탐색 불가(NPE 위험)
```
- (A): 외래 키(member_id)를 객체 참조로 만들려고 회원을 따로 조회·연결하는 '수동 조립' 필요. ✅
- (B): 같은 DB 행(PK=1)이어도 자바 객체는 `==` false. ✅
- (C): 회원을 안 채운 Order는 member=null이라 `order.getMember().getName()`로 더 탐색하면 NPE. SQL이 채운
  만큼만 객체가 차 있다(탐색 한계). ✅ → ORM이 (A)(B)(C)를 자동화(조립·동일 PK 관리·지연 로딩)해 메운다(11.3~).

## 3. 자기 점검

- **Q. 객체-관계 미스매치 측면들을 들면?**
  - 내 답: 모델링(상태+행동 vs 행·열), 상속(있음 vs 없음), 연관(참조 vs 외래 키), 식별(== vs PK),
    탐색(자유 탐색 vs JOIN 범위), 세분성(값 객체로 잘게 vs 컬럼으로 펼침).

- **Q. 탐색(navigation) 미스매치란? 무엇으로 이어지나?**
  - 내 답: 객체는 참조를 자유롭게 타고 들어가지만, SQL Mapper는 그 SQL이 조회한 범위까지만 객체를 채운다.
    조회 안 한 연관은 null이라 더 탐색하면 NPE. "어디까지 채워졌나"가 SQL마다 달라 객체 그래프를 신뢰할 수
    없다. 이 한계가 JPA 지연 로딩과 '신뢰할 수 있는 객체 그래프'의 동기다. (Main의 C)

- **Q. 세분성(granularity) 미스매치란?**
  - 내 답: 객체는 Address 같은 값 객체로 더 잘게 표현하지만 DB는 컬럼 몇 개로 펼친다. 표현 단위가 달라
    '객체 한 덩어리 ↔ 컬럼 여러 개' 변환이 필요하다. JPA는 @Embedded 값 타입으로 매핑한다.

- **Q. '객체 그래프 수동 조립'이 왜 고통인가?**
  - 내 답: DB는 외래 키(숫자)만 주므로, 참조 객체를 따로 조회해 new 하고 연결하는 변환 코드를 쿼리마다
    반복해야 한다. ORM은 이를 자동(연관 따라 조회/연결)으로 처리한다.

- **Q. 같은 PK인데 == 가 false인 식별 미스매치는 ORM이 어떻게 푸나?**
  - 내 답: 영속성 컨텍스트(1차 캐시)로 "같은 PK는 같은 객체 인스턴스"를 반환해 동일성을 보장한다(11.3~).

- **Q. ORM의 단점은?**
  - 내 답: 학습곡선이 높고, 연관을 무심코 조회하면 N+1 쿼리가 터지며, 생성 SQL이 숨겨져 튜닝이 어렵다.
    그래서 복잡 쿼리는 SQL/Querydsl을 병행한다.
