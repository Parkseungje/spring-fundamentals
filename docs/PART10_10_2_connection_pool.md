# PART 10 — DB 접근의 진화: 10.2 Connection Pool

> 이 문서는 커리큘럼 PART 10의 소단원 중 **10.2 Connection Pool**을 다룬다.
> 10.1에서 JDBC로 'DB에 어떻게 접근하나'를 표준화했다면, 10.2는 '연결을 매번 새로 만드는 비효율'을
> 연결 재사용(풀)으로 푼다. (실무 직결)

---

## 0. 들어가기 전에 — 핵심 용어
- **Connection(커넥션)**: 애플리케이션과 DB 사이의 연결 통로. 만들 때 TCP 연결 + 인증 + 세션 생성 비용이 든다.
- **Connection Pool(커넥션 풀)**: 연결을 미리 N개 만들어 두고 빌려주고(borrow)·반납(return)받아 '재사용'하는 저장소.
- **HikariCP**: Spring Boot의 기본 커넥션 풀 구현. (다른 구현: DBCP2, Tomcat JDBC Pool)
- **DB 세션(session)**: DB 입장에서 본 하나의 연결 단위. 풀의 Connection N개 = DB 세션 N개.
- **DB Lock(락)**: 트랜잭션 진행 중 다른 세션의 수정을 막는 잠금. 공유 락(S, 읽기) / 배타 락(X, 쓰기). DB에서도 데드락이 생긴다.

한 줄 그림: **연결을 매 요청마다 새로 만들면 '연결 비용'이 쿼리보다 더 들 수 있다. 미리 만들어 둔 연결을
빌려 쓰고 반납하는 '풀'을 쓰면 재사용으로 빠르다. Spring Boot는 HikariCP를 기본으로 쓴다.**

---

## 1. 학습 내용 — 매번 연결 vs 재사용

### 로우레벨의 비효율 — 연결은 비싸다
DB 연결 1회는 단순히 "선 하나 잇기"가 아니다. **TCP 3-way handshake + 인증(계정 확인) + DB 세션 생성**이
일어난다. 실제 원격 DB(MySQL/Oracle)에선 수~수백 ms가 걸리기도 한다. 그래서 매 요청마다 새 연결을 만들면
**연결 설정 시간이 정작 쿼리 실행 시간보다 더 길어지는** 어이없는 상황이 생긴다.

### 해결 — Connection Pool ('커넥션의 수영장')
- 애플리케이션 시작 시 연결을 **N개 미리 생성**해 풀에 담아 둔다.
- 요청이 오면 풀에서 하나 **빌리고(borrow)**, 다 쓰면 닫지 않고 풀에 **반납(return)**한다.
- 다음 요청은 그 연결을 **재사용** → 매번 생성 비용을 안 치른다. 동시 연결 수도 풀 크기로 제한돼 안정적.
- 대표 구현: **HikariCP(Spring Boot 기본)**, DBCP2, Tomcat JDBC Pool.
```java
HikariConfig cfg = new HikariConfig();
cfg.setJdbcUrl(url); cfg.setUsername("sa"); cfg.setMaximumPoolSize(10);
HikariDataSource pool = new HikariDataSource(cfg);
try (Connection c = pool.getConnection()) { ... }   // 빌림 -> try 종료 시 '반납'(close=반환)
```
> ★ 포인트: 풀에서 얻은 Connection의 `close()`는 '연결을 끊는' 게 아니라 '풀에 반납하는' 것이다. 그래서
> close해도 다음에 또 빌려 쓸 수 있다(PART 8.6 try-with-resources와 같은 모양이지만 의미는 '반납').

### ★ "커넥션 풀 10 = 동시 접속 10명?" — 아니다 (동시 DB 작업 10개)
커넥션 풀 크기 10은 "접속한 사용자 10명"이 아니라 **"그 순간 실제로 DB와 통신(쿼리) 중일 수 있는 게
최대 10개"** 다(한 WAS 기준).
- 사용자 1000명이 사이트를 보고 있어도, 대부분은 그 찰나에 DB를 안 건드린다(화면 보는 중 등). **DB 쿼리를
  날리는 요청만** 커넥션을 빌린다. 웹 요청은 보통 수십 ms 만에 빌렸다 반납하므로, 10개로도 초당 수백~수천 요청 처리 가능.
- **11번째 동시 요청**은 "안 됨"이 아니라 **빈 커넥션이 반납될 때까지 대기**한다. 대기가 너무 길면 HikariCP가
  타임아웃(기본 30초)을 낸다(`Connection is not available`) → 이것이 '커넥션 풀 고갈' 장애다.

**풀 크기 ≠ DB의 max_connections** (별개의 한도):
- 커넥션 풀 크기(10) = *내 WAS 한 대*가 DB에 들 수 있는 연결 수(내가 정함).
- DB의 max_connections = *DB 서버 전체*가 받을 수 있는 연결 수(예: MySQL 기본 151).
- WAS가 3대고 각자 풀 10이면 DB엔 최대 30개 연결 → **풀 크기 × WAS 대수 ≤ DB max_connections**를 고려해야 한다.

**풀은 클수록 좋은 게 아니다**: DB가 동시에 처리할 수 있는 일은 CPU·디스크에 한계가 있어, 커넥션이 너무
많으면 DB가 컨텍스트 스위칭·락 경합으로 오히려 느려진다(PART 7.1 "스레드 많다고 빠르지 않다"와 같은 맥락).
HikariCP 권장도 보통 코어 수 기반의 작은 값(수십 개)이다.

비유: 커넥션 = 은행 창구 10개. 손님 1000명이 와 있어도 창구에서 실제 업무 중인 건 최대 10명, 나머지는
둘러보거나 대기. 손님이 금방 처리되면 10개로 하루 수천 명 처리. 창구를 100개로 늘려도 직원(DB CPU)이 감당 못 하면 오히려 느려진다.

### ★ 커넥션은 '행(row)'과 무관하다 — 락과 별개 층위
"커넥션이 같은 행에만 해당되나?"는 좋은 질문이다. **아니다. 커넥션은 행과 완전히 무관**하다. 락과 커넥션은
서로 다른 층위다.
- **커넥션(풀)**: "DB에 들어가는 통로/출입증". 어느 행을 건드리든 **DB 작업(트랜잭션) 하나당 통로 하나**를 차지. 개수 문제.
- **락(row-level)**: "**같은 행**을 동시에 수정할 때만" 작동. 충돌 문제.

10명이 **각자 다른 행**을 동시에 수정해도:
```
10명 동시 트랜잭션 (id=1,2,...,10, 다 다른 행)
 → 커넥션 10개를 각각 하나씩 차지 (통로 10개 다 씀)
 → 락 충돌? 없음(다 다른 행) → 10개 동시 처리 ✅
 → 11번째 요청(또 다른 행 id=11)은? 커넥션이 없어 대기 (락 때문이 아니라 '통로'가 없어서 = 풀 고갈)
```

| 상황 | 커넥션 | 락 |
|---|---|---|
| 10명이 각자 다른 행 수정 | 10개 사용(각 1개) | 충돌 없음 → 동시 진행 |
| 10명이 같은 행 수정 | 10개 사용(각 1개) | 같은 행 락 → 한 명씩 순차 |

→ **둘 다 커넥션은 10개 똑같이 차지**한다(행 무관). 차이는 '락 때문에 서로 기다리느냐'뿐. 즉 커넥션을
차지하는 기준 = "DB 작업 중이냐"(행 무관), 락이 걸리는 기준 = "같은 행을 동시에 건드리느냐".

### 풀과 DB 세션·락 (개념)
- **풀의 Connection N개 = DB 세션 N개.** 한 Connection(세션) 안에서 **동시에 두 트랜잭션은 불가**하다
  (트랜잭션은 연결 단위 — 10.4와 연결).
- **DB Lock**: 한 트랜잭션이 행을 수정 중이면 다른 세션의 수정을 막는다(배타 락 X). 읽기는 공유 락(S).
  자바의 synchronized/락과 닮은꼴이고, **DB에서도 데드락이 생긴다**(PART 7.6의 데드락과 같은 원리, 무대가 DB일 뿐).

#### ★ 오해 바로잡기 — "락 때문에 동시 수정이 안 된다"? (아니다)
"같은 DB를 쓰면 두 사용자가 동시에 글을 못 고치나?" 하는 오해가 흔하다. **아니다. 동시 수정 잘 된다.**
- **락은 '같은 행(row)'을 동시에 건드릴 때만** 작동한다. 락은 **행 단위(row-level)** — 테이블/DB 전체가
  아니라 '수정 중인 그 행 하나'만 잠근다.
  ```
  나:   update post set ... where id=10   (내 글)
  친구: update post set ... where id=77   (친구 글)
  -> id가 달라 락 충돌 없음 -> 둘 다 동시에 즉시 수정 ✅
  ```
- **'같은 행'을 동시에 수정할 때조차** "안 됨"이 아니라 "잠깐 기다렸다 순서대로"다. 한쪽이 commit/rollback할
  때까지 다른 쪽이 밀리초 단위로 대기했다가 처리된다(영영 막히는 게 아님).
- **"한 Connection=한 트랜잭션"은 서버 내부 규칙**이지 사용자 수 제한이 아니다. 사용자가 둘이면 서버가
  커넥션을 둘 빌려(나→커넥션#1, 친구→커넥션#2) 동시에 처리한다.

#### ★ 그럼 둘이 '같은 글'을 거의 동시에 수정하면? — Last Write Wins + 갱신 유실
같은 글(id=10)을 나(T0.0)와 친구(T0.1)가 거의 동시에 수정하면:
```
1) 내 트랜잭션이 id=10에 배타 락 -> '내글'로 수정
2) 친구 트랜잭션은 같은 행이라 락이 풀릴 때까지 '잠깐 대기'
3) 내가 commit -> 락 해제 -> '내글' 확정
4) 친구가 락을 얻어 '친구글'로 덮어씀 -> commit -> 최종 '친구글'
```
- 결과: 새로고침하면 **나중에 commit한 친구 글**이 보인다. **순차 처리 + 나중 것이 덮어씀(Last Write Wins).**
- ★ 중요: **락은 "데이터가 뒤섞여 깨지는 것"은 막았지만(무결성), "내 수정이 조용히 사라지는 것"은 못 막았다**
  (갱신 유실 lost update — PART 7.4와 같은 개념, 무대가 DB). 락은 '순서 정렬'이지 '사용자 의도 충돌 감지'가 아니다.

#### 이 유실을 막으려면 — 낙관적 락(@Version)
"내가 본 글이 그새 바뀌었으면 덮어쓰지 말고 알려줘"가 필요하면 버전 컬럼을 둔다(JPA `@Version` — 이후 PART).
```
나:   update post set content='내글',  version=2 where id=10 and version=1  -> 성공(1->2)
친구: update post set content='친구글', version=2 where id=10 and version=1  -> 0건 수정!(version이 이미 2)
      -> 앱이 "다른 사람이 먼저 수정했습니다" 알림으로 친구의 덮어쓰기를 막음
```
위키·구글독스의 "다른 사람이 편집했습니다"가 이 원리다. (DB 기본 락 = 비관적 관점, @Version = 낙관적 관점)

### 커넥션 풀 vs 스레드 풀 (같은 '풀 패턴', 다른 대상·목적)
"커넥션 풀과 스레드 풀(PART 7.8)이 비슷한가?"는 좋은 질문이다. **풀이라는 패턴(미리 만들어 빌리고·반납·재사용)은
똑같고, 담는 자원과 푸는 문제가 다르다.**

| | 스레드 풀 (PART 7.8) | 커넥션 풀 (여기, 10.2) |
|---|---|---|
| 담는 자원 | 스레드(실행 흐름) | DB 커넥션(DB 연결) |
| 비싼 이유 | 생성 시 스택 ~1MB + OS 등록 | TCP 핸드셰이크 + 인증 + 세션 생성 |
| 푸는 문제 | "작업을 누가 실행하나"(동시 처리) | "DB에 어떻게 연결하나"(DB 접근) |
| 대표 구현 | `ExecutorService`(newFixedThreadPool) | HikariCP |
| 크기 의미 | 동시 실행 작업 수 | 동시 DB 세션 수 |

**실무에선 둘이 협력한다.** 웹 요청 처리 흐름:
```
요청 → [스레드 풀에서 스레드 1개 빌림] → 그 스레드가 비즈니스 로직 실행
          → DB 필요 → [커넥션 풀에서 커넥션 1개 빌림] → 쿼리 → 커넥션 반납
       → 작업 끝 → 스레드 반납
```
- **크기 균형이 중요**: 스레드 풀 200 + 커넥션 풀 10이면, 동시 요청 200개가 와도 DB 작업은 한 번에 10개만
  가능 → 나머지는 커넥션을 기다린다(병목). "스레드 풀이 크다고 처리량이 비례해 늘지 않는다"(PART 7.1
  "스레드 많다고 무조건 빠르지 않다"와 같은 맥락) — 커넥션 풀/DB가 병목일 수 있다.
- 비유: 스레드 풀 = 식당 '종업원' 풀, 커넥션 풀 = 주방(DB) 출입증 풀. 종업원이 20명이어도 출입증이 5개면
  한 번에 5명만 주방에 들어간다.

## 3. 자기 점검

- **Q. 커넥션 풀이 빠른 이유는?**
  - 내 답: 연결(TCP 핸드셰이크+인증+세션 생성)을 미리 만들어 두고 재사용하므로, 매 요청마다 치르던 연결
    생성 비용을 없애기 때문. (실측 336ms→39ms)

- **Q. 풀에서 얻은 Connection을 close()하면 연결이 끊기나?**
  - 내 답: 아니다. '풀에 반납'된다. 그래서 다음 요청이 그 연결을 또 빌려 쓴다(재사용).

- **Q. 풀 크기와 DB 세션의 관계는?**
  - 내 답: 풀의 Connection N개 = DB 세션 N개. 한 연결(세션) 안에서 동시에 두 트랜잭션은 못 돌린다.
    그래서 동시 처리량과 DB 부하를 풀 크기로 조절한다.

- **Q. DB Lock/데드락은 PART 7과 무슨 관계인가?**
  - 내 답: 개념이 같다. 한 트랜잭션이 행을 잠그면 다른 세션이 기다리고, 서로 엇갈려 잠그면 데드락이 난다
    (7.6의 락·데드락과 동일 원리, 무대가 JVM이 아니라 DB일 뿐).

- **Q. 커넥션 풀과 스레드 풀(7.8)은 같은 건가?**
  - 내 답: '풀 패턴'(미리 만들어 빌리고·반납·재사용)은 같지만 대상·목적이 다르다. 스레드 풀=스레드(동시
    처리), 커넥션 풀=DB 커넥션(DB 접근). 실무에선 협력한다 — 요청 스레드가 커넥션을 빌려 DB 작업.
    스레드 풀이 커도 커넥션 풀이 작으면 거기서 병목이 생긴다(크기 균형 중요).

- **Q. 두 사용자가 같은 DB의 '다른 글'을 동시에 수정하면 막히나?**
  - 내 답: 안 막힌다. 락은 '같은 행'을 동시에 건드릴 때만 작동하는 행 단위 락이라, 다른 글(다른 행)은
    서로 무관하게 동시에 수정된다. 사용자가 둘이면 커넥션도 둘이라 동시에 처리된다.

- **Q. 두 사용자가 '같은 글'을 거의 동시에 수정하면 결과는?**
  - 내 답: 락으로 순차 처리되어, 나중에 commit한 쪽이 덮어쓴다(Last Write Wins). 데이터가 깨지진 않지만
    먼저 쓴 사람의 수정이 조용히 유실된다(lost update). 막으려면 낙관적 락(@Version)으로 "버전이 바뀌었으면
    덮어쓰기 거부"를 한다.

- **Q. 커넥션 풀 10이면 'DB 동시 접속 10명'인가?**
  - 내 답: 아니다. "그 순간 동시에 DB 작업 중일 수 있는 게 최대 10개"(한 WAS 기준)다. 접속자 1000명이어도
    DB 쿼리 중인 요청만 커넥션을 빌리고 금방 반납한다. 11번째는 대기(너무 길면 타임아웃=풀 고갈). 또 풀 크기는
    DB의 max_connections(DB 전체 한도)와 별개라, WAS 여러 대면 합산이 DB 한도를 넘지 않게 해야 한다.

- **Q. 커넥션은 같은 행을 수정할 때만 차지하나?**
  - 내 답: 아니다. 커넥션은 행과 무관하게 'DB 작업(트랜잭션) 하나당 통로 하나'를 차지한다. 10명이 각자
    다른 행을 동시에 수정해도 커넥션 10개를 다 쓴다(락 충돌은 없지만). 락(같은 행)과 커넥션(통로 개수)은 별개 층위.
