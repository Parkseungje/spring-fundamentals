# PART 10 — DB 접근의 진화: 10.2 Connection Pool

> 이 문서는 커리큘럼 PART 10의 소단원 중 **10.2 Connection Pool**을 다룬다.
> 10.1에서 JDBC로 'DB에 어떻게 접근하나'를 표준화했다면, 10.2는 '연결을 매번 새로 만드는 비효율'을
> 연결 재사용(풀)으로 푼다. (실무 직결)

---

## 0. 들어가기 전에 — 핵심 용어
- **Connection(커넥션)**: 애플리케이션과 DB 사이의, 이미 열려서 바로 쓸 수 있는 '통신 선'(TCP 연결 + 인증된 DB 세션).
- **Connection Pool(커넥션 풀)**: 그 커넥션을 미리 N개 만들어 두고 빌려주고(borrow)·반납(return)받아 '재사용'하는 저장소.
- **HikariCP**: Spring Boot의 기본 커넥션 풀 구현. (다른 구현: DBCP2, Tomcat JDBC Pool)
- **DB 세션(session)**: DB 입장에서 본 하나의 연결 단위. 풀의 Connection N개 = DB 세션 N개.
- **max_connections**: DB 서버 전체가 받을 수 있는 연결 수(풀 크기와 다른 한도).
- **DB Lock(락)**: 같은 행을 동시에 수정하려 할 때 작동하는 잠금(행 단위). 커넥션과는 별개 층위.

한 줄 그림: **커넥션(TCP+인증+세션을 마친 통신 선)을 매 요청마다 새로 만들면 너무 비싸다. 그래서 미리
만들어 두고 빌려 쓰고 반납하는 '풀'로 재사용한다(Spring Boot 기본 HikariCP). 풀 크기 = '동시에 DB 작업
중일 수 있는 수'다.**

---

## 1. 학습 내용

### 1-1. 커넥션이란 무엇인가 — 풀이 '무엇을' 미리 만들어 두나
`DriverManager.getConnection(url, ...)` 한 줄 뒤에는 무거운 작업이 숨어 있다.
```
커넥션 1개 생성 =
  ① TCP 3-way handshake   (앱 ↔ DB 네트워크 연결 수립)
  ② 인증(authentication)   (계정/비번 확인, 권한 체크)
  ③ DB 세션 생성           (DB가 이 연결 전용 메모리·상태 공간 할당)
  ④ 프로토콜 협상           (SSL이면 암호화 키 교환까지)
```
이게 끝나면 결과물은 **`Connection` 객체 하나** — "DB와 이미 연결되어 인증까지 끝난, 바로 SQL을 보낼 수
있는 통신 선"이다. 원격 DB면 ①~④에 **수~수백 ms**가 걸린다.

→ **풀이 미리 만들어 두는 것이 바로 이 'Connection 객체(열린 통신 선) N개'다.** 앱 시작 시 ①~④를 미리
N번 수행해 열린 커넥션들을 담아 둔다.

#### ③ 'DB 세션 생성'이란 — 디스크가 아니라 'DB 서버 RAM'에 잡는 공간
위 ③의 "세션 생성"이 헷갈리기 쉽다. 이건 **MySQL이 설치된 디렉토리(디스크)에 뭘 쓰는 게 아니라, 실행
중인 MySQL 서버 프로세스의 메모리(RAM)** 에 그 연결 전용 공간을 잡는 것이다.
```
[MySQL 서버 = 실행 중인 프로세스 (RAM)]
  ├─ 공용 메모리(버퍼 풀 등 — 모든 연결이 공유)
  ├─ 세션1 전용 메모리  ← 커넥션1 연결 시 할당
  ├─ 세션2 전용 메모리  ← 커넥션2 연결 시 할당
  └─ ...
```
- 세션 전용 메모리에 담기는 것: 그 연결의 트랜잭션 상태(auto-commit 등), 세션 변수(문자셋·타임존·격리 수준),
  쿼리 작업 버퍼(정렬·조인용), 준비된 statement·임시 테이블 등. "이 연결이 지금 뭘 하는지"의 상태·작업 공간이다.
- **양쪽에 메모리가 잡힌다**: DB 서버(MySQL) RAM의 세션 + 내 앱(JVM) RAM의 `Connection` 객체·소켓 버퍼. 둘이 짝.
- **데이터(행)는 디스크, 세션은 RAM**: 실제 테이블 데이터는 디스크 데이터 파일(`.ibd` 등)에 영구 저장(commit 시
  fsync — 10.4 지속성). 세션/연결은 RAM이라 연결이 끊기면 사라진다. (식당 비유: 데이터 파일=창고의 재료,
  세션=손님 한 팀을 위한 테이블 세팅. 세팅은 손님이 있는 동안만.)

#### 풀 N개 = MySQL 서버 RAM에 세션 N개가 (앱 떠 있는 동안) 상주
풀이 커넥션 10개를 만든다 = **실제로 MySQL에 10번 연결을 맺는다** = MySQL RAM에 **세션 10개가 생긴다**.
- 이 10개는 앱이 떠 있는 **내내 살아있다**(반납해도 끊지 않고 풀에 보관하므로 MySQL 세션도 유지). 쿼리를
  안 날려도 MySQL엔 그 10개가 Sleep 상태로 떠 있다(`SHOW PROCESSLIST`로 확인 가능).
  (HikariCP 기본은 minimumIdle = maximumPoolSize라 보통 풀 크기만큼 처음부터 유지.)
- 정밀하게 나누면: **세션의 기본 골격(연결·세션 변수 등)은 미리 상주**, **무거운 쿼리용 작업 버퍼(sort/join)는
  쿼리할 때 그때 추가**로 잡혔다 풀린다. 그래서 세션당 최대 메모리 ≈ 기본 + 쿼리 시 버퍼.
- 그래서 **연결 수 = DB가 상주시켜야 할 세션 메모리**다. MySQL 메모리 ≈ 공용 메모리 + 세션당 메모리 × 연결 수.
  WAS 3대 × 풀 10 = MySQL 세션 30개 = 그만큼의 RAM. → 이것이 1-5의 "풀을 무작정 키우면 안 되는" 직접적 이유다.

### 1-2. 로우레벨의 비효율 — 매번 만들면 연결이 쿼리보다 비싸다
매 요청마다 커넥션을 새로 만들면, 정작 쿼리는 수 ms인데 **연결 설정(①~④)이 수십~수백 ms**라 배보다
배꼽이 커진다. 게다가 동시 요청이 몰리면 연결 생성이 폭증해 DB가 휘청인다.

### 1-3. 해결 — Connection Pool로 '재사용'
```
[풀 없음]  요청 → ①②③④ 연결 생성(수십~수백 ms) → 쿼리(수 ms) → 연결 끊기   (매번 반복)
[풀 있음]  요청 → 풀에서 '이미 열린' 커넥션 빌림(즉시) → 쿼리 → 풀에 반납       (①②③④ 생략!)
```
- 앱 시작 시 커넥션을 **N개 미리 생성**해 풀에 담는다.
- 요청이 오면 **빌리고(borrow)**, 다 쓰면 **반납(return)**한다(다음 요청이 또 씀). → 매번 생성 비용을 안 치르니 빠르다.
```java
HikariConfig cfg = new HikariConfig();
cfg.setJdbcUrl(url); cfg.setUsername("sa"); cfg.setMaximumPoolSize(10);
HikariDataSource pool = new HikariDataSource(cfg);
try (Connection c = pool.getConnection()) { ... }   // 빌림 -> try 종료 시 '반납'
```
> ★ 핵심: 풀에서 얻은 Connection의 `close()`는 **연결을 끊는 게 아니라 '풀에 반납'**하는 것이다. TCP 선과
> DB 세션은 계속 살아 있어, 다음 요청이 ①~④ 없이 즉시 다시 빌려 쓴다. (PART 8.6 try-with-resources와 같은
> 모양이지만 의미는 '끊기'가 아니라 '반납'.)
>
> 비유: 전화. 커넥션 생성 = 전화 걸어 상대가 받고 본인 확인까지 끝낸 상태(①~④). 풀 = 이미 통화 연결된
> 전화선 10개를 깔아 둠. 빌림 = 새로 다이얼하지 않고 연결된 전화를 들어 용건만 말함. 반납 = 안 끊고 보류했다 다음 사람이 또 씀.

### 1-4. 풀 크기의 의미 — "동시 접속 10명"이 아니다
커넥션 풀 크기 10은 "접속한 사용자 10명"이 아니라 **"그 순간 실제로 DB와 통신(쿼리) 중일 수 있는 게 최대
10개"** 다(한 WAS 기준).
- 사용자 1000명이 사이트를 보고 있어도, 대부분은 그 찰나에 DB를 안 건드린다(화면 보는 중 등). **DB 쿼리를
  날리는 요청만** 커넥션을 빌리고, 보통 수십 ms 만에 반납한다. 그래서 10개로도 초당 수백~수천 요청을 처리한다.
- **11번째 동시 요청**은 "안 됨"이 아니라 **빈 커넥션이 반납될 때까지 대기**한다. 대기가 너무 길면 HikariCP가
  타임아웃(기본 30초)을 낸다(`Connection is not available`) → 이것이 '커넥션 풀 고갈' 장애다.

비유: 커넥션 = 은행 창구 10개. 손님 1000명이 와 있어도 창구에서 실제 업무 중인 건 최대 10명, 나머지는
둘러보거나 대기. 손님이 금방 처리되면 10개로 하루 수천 명을 처리한다.

### 1-5. 풀 크기 ≠ DB의 max_connections, 그리고 적정 크기
- **풀 크기**(예: 10) = *내 WAS 한 대*가 DB에 들 수 있는 연결 수(내가 정함).
- **max_connections**(예: MySQL 기본 151) = *DB 서버 전체*가 받을 수 있는 연결 수.
- WAS가 3대고 각자 풀 10이면 DB엔 최대 30개 연결 → **풀 크기 × WAS 대수 ≤ DB max_connections**여야 한다.
- **풀은 클수록 좋은 게 아니다**: DB의 동시 처리 능력은 CPU·디스크에 한계가 있어, 커넥션이 너무 많으면
  DB가 컨텍스트 스위칭·락 경합으로 오히려 느려진다(PART 7.1 "스레드 많다고 빠르지 않다"와 같은 맥락).
  HikariCP 권장도 보통 코어 수 기반의 작은 값(수십 개)이다. (창구를 100개로 늘려도 직원=DB CPU이 감당 못 하면 더 느려짐.)

### 1-6. 커넥션은 '행(row)'과 무관하다 — 락과는 별개 층위
커넥션과 락은 서로 다른 층위다. 헷갈리지 말자.
- **커넥션(풀)**: "DB에 들어가는 통로". 어느 행을 건드리든 **DB 작업(트랜잭션) 하나당 통로 하나**를 차지(개수 문제).
- **락(row-level)**: "**같은 행**을 동시에 수정할 때만" 작동(충돌 문제).

| 상황 | 커넥션 | 락 |
|---|---|---|
| 10명이 각자 다른 행 수정 | 10개 사용(각 1개) | 충돌 없음 → 동시 진행 |
| 10명이 같은 행 수정 | 10개 사용(각 1개) | 같은 행 락 → 한 명씩 순차 |

→ 행이 같든 다르든 **커넥션은 동시 작업 수만큼(여기선 10개) 똑같이 차지**한다. 11번째 요청은 (행과 무관하게)
커넥션이 없어서 대기한다(락이 아니라 '통로 부족' = 풀 고갈). 즉 **커넥션 차지 기준 = "DB 작업 중이냐"(행 무관)**,
락 작동 기준 = "같은 행을 동시에 건드리느냐".

### 1-7. 풀과 DB 세션·락, 그리고 동시 수정
- **풀의 Connection N개 = DB 세션 N개.** 한 Connection(세션) 안에서 **동시에 두 트랜잭션은 불가**(트랜잭션은 연결 단위 — 10.4).
  단 "한 커넥션=한 트랜잭션"은 서버 내부 규칙이지 사용자 수 제한이 아니다(사용자 둘이면 커넥션도 둘 빌림).
- **DB Lock**: 같은 행을 수정 중인 트랜잭션이 있으면 다른 세션의 같은 행 수정을 막는다(배타 락 X, 읽기는 공유 락 S).
  자바 synchronized와 닮은꼴이고 **DB에서도 데드락이 생긴다**(PART 7.6과 같은 원리, 무대가 DB).

**두 사용자가 '다른 글'을 동시 수정 → 막힘 없음**(다른 행이라 락 충돌 0, 커넥션도 각자 빌림).
**두 사용자가 '같은 글'을 거의 동시 수정 → 순차 처리 + 나중 것이 덮어씀(Last Write Wins)**:
```
1) 내 트랜잭션이 id=10에 배타 락 -> '내글' 수정
2) 친구 트랜잭션은 같은 행이라 락 풀릴 때까지 잠깐 대기
3) 내가 commit -> 락 해제 -> '내글' 확정
4) 친구가 락 얻어 '친구글'로 덮어씀 -> commit -> 최종 '친구글'
```
- ★ 락은 "데이터가 뒤섞여 깨지는 것"은 막지만(무결성), "내 수정이 조용히 사라지는 것"은 못 막는다
  (갱신 유실 lost update — PART 7.4와 같은 개념). 락은 '순서 정렬'이지 '사용자 의도 충돌 감지'가 아니다.
- 막으려면 **낙관적 락(@Version)**: `update ... where id=10 and version=1` → 버전이 이미 바뀌었으면 0건
  수정 → 앱이 "다른 사람이 먼저 수정했습니다" 알림(위키·구글독스 원리). JPA `@Version`(이후 PART).

### 1-8. 커넥션 풀 vs 스레드 풀 — 같은 패턴, 다른 대상
| | 스레드 풀 (PART 7.8) | 커넥션 풀 (여기) |
|---|---|---|
| 담는 자원 | 스레드(실행 흐름) | DB 커넥션(통신 선) |
| 비싼 이유 | 스택 ~1MB + OS 등록 | TCP+인증+세션 생성 |
| 푸는 문제 | "작업을 누가 실행하나" | "DB에 어떻게 연결하나" |
| 크기 의미 | 동시 실행 작업 수 | 동시 DB 작업 수 |

**실무에선 협력한다**: 요청 → 스레드 풀에서 스레드 빌림 → 그 스레드가 커넥션 풀에서 커넥션 빌려 DB 작업.
스레드 풀 200 + 커넥션 풀 10이면, 동시 요청 200개여도 DB 작업은 한 번에 10개 → 나머지는 커넥션 대기(병목).
"스레드 많다고 처리량이 비례해 늘지 않는다"(PART 7.1). 비유: 종업원(스레드) 풀 + 주방 출입증(커넥션) 풀.

## 3. 자기 점검

- **Q. 풀이 '미리 만들어 두는' 커넥션이란 무엇인가?**
  - 내 답: TCP 연결 + 인증 + DB 세션 생성(①~④)을 마친, 바로 SQL을 보낼 수 있는 'Connection 객체(열린 통신
    선)'다. 그 무거운 생성을 미리 N번 해두고 재사용한다.

- **Q. 커넥션 풀이 빠른 이유와, close()의 의미는?**
  - 내 답: 매 요청마다 치르던 연결 생성 비용(TCP+인증+세션)을 안 치르고 이미 열린 걸 빌려 쓰기 때문. 풀에서
    얻은 close()는 끊기가 아니라 '풀에 반납'이라, 다음 요청이 또 재사용한다. (실측 336→39ms)

- **Q. 커넥션 풀 10이면 'DB 동시 접속 10명'인가?**
  - 내 답: 아니다. "그 순간 동시에 DB 작업 중일 수 있는 게 최대 10개"(한 WAS 기준). 접속자 1000명이어도 DB
    쿼리 중인 요청만 빌리고 금방 반납한다. 11번째는 대기(너무 길면 타임아웃=풀 고갈). 풀 크기는 DB의
    max_connections(DB 전체 한도)와 별개라, WAS 여러 대면 합산이 DB 한도를 넘지 않게 한다.

- **Q. 커넥션은 같은 행을 수정할 때만 차지하나?**
  - 내 답: 아니다. 행과 무관하게 'DB 작업(트랜잭션) 하나당 통로 하나'를 차지한다. 10명이 각자 다른 행을
    수정해도 커넥션 10개를 다 쓴다. 락(같은 행)과 커넥션(통로 개수)은 별개 층위.

- **Q. 두 사용자가 '같은 글'을 거의 동시에 수정하면?**
  - 내 답: 락으로 순차 처리되어 나중에 commit한 쪽이 덮어쓴다(Last Write Wins). 데이터는 안 깨지지만 먼저 쓴
    수정이 조용히 유실된다(lost update). 막으려면 낙관적 락(@Version)으로 "버전이 바뀌었으면 덮어쓰기 거부".

- **Q. 커넥션 풀과 스레드 풀(7.8)은 같은 건가?**
  - 내 답: '풀 패턴'(미리 만들어 빌리고·반납·재사용)은 같지만 대상·목적이 다르다(커넥션=DB 연결, 스레드=실행
    흐름). 실무에선 협력하며, 스레드 풀이 커도 커넥션 풀이 작으면 거기서 병목이 생긴다.

- **Q. 'DB 세션 생성'은 MySQL 설치 디렉토리(디스크)에 잡는 건가?**
  - 내 답: 아니다. 실행 중인 MySQL 프로세스의 RAM에 그 연결 전용 공간(트랜잭션 상태·세션 변수·작업 버퍼)을
    잡는 것이다. 데이터(행)는 디스크 데이터 파일에 영구 저장되지만, 세션/연결은 RAM(휘발성)이다.

- **Q. 풀 10이면 MySQL은 그 10개 세션 메모리를 미리 잡아두나?**
  - 내 답: 그렇다. 풀 10 = MySQL에 실제 연결 10개 = MySQL RAM에 세션 10개가 앱 떠 있는 동안 상주(Sleep
    상태로 유지). 세션 기본 골격은 미리, 무거운 쿼리용 버퍼는 쿼리할 때 추가로 잡힌다. 그래서 연결 수가
    DB RAM을 직접 좌우한다(풀을 무작정 못 키우는 이유).
