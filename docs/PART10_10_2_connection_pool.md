# PART 10 — DB 접근의 진화: 10.2 Connection Pool

> 이 문서는 커리큘럼 PART 10의 소단원 중 **10.2 Connection Pool**을 다룬다.
> 10.1에서 JDBC로 'DB에 어떻게 접근하나'를 표준화했다면, 10.2는 '연결을 매번 새로 만드는 비효율'을
> 연결 재사용(풀)으로 푼다. (실무 직결)

---

## 0. 들어가기 전에 — 핵심 용어
- **Connection(커넥션)**: 애플리케이션과 DB 사이의, 이미 열려서 바로 쓸 수 있는 '통신 선'(TCP 연결 + 인증된 DB 세션).
- **Connection Pool(커넥션 풀)**: 그 커넥션을 미리 N개 만들어 두고 빌려주고(borrow)·반납(return)받아 '재사용'하는 저장소.
- **HikariCP**: Spring Boot의 기본 커넥션 풀 구현. (다른 구현: DBCP2, Tomcat JDBC Pool)
- **DB 세션(session)**: DB 입장에서 본 하나의 연결 단위. 풀의 Connection N개 = DB 세션 N개.
- **max_connections**: DB 서버 전체가 받을 수 있는 연결 수(풀 크기와 다른 한도).
- **DB Lock(락)**: 같은 행을 동시에 수정하려 할 때 작동하는 잠금(행 단위). 커넥션과는 별개 층위.

한 줄 그림: **커넥션(TCP+인증+세션을 마친 통신 선)을 매 요청마다 새로 만들면 너무 비싸다. 그래서 미리
만들어 두고 빌려 쓰고 반납하는 '풀'로 재사용한다(Spring Boot 기본 HikariCP). 풀 크기 = '동시에 DB 작업
중일 수 있는 수'다.**

---

## 1. 학습 내용

### 1-1. 커넥션이란 무엇인가 — 풀이 '무엇을' 미리 만들어 두나
`DriverManager.getConnection(url, ...)` 한 줄 뒤에는 무거운 작업이 숨어 있다.
```
커넥션 1개 생성 =
  ① TCP 3-way handshake   (앱 ↔ DB 네트워크 연결 수립)
  ② 인증(authentication)   (계정/비번 확인, 권한 체크)
  ③ DB 세션 생성           (DB가 이 연결 전용 메모리·상태 공간 할당)
  ④ 프로토콜 협상           (SSL이면 암호화 키 교환까지)
```
이게 끝나면 결과물은 **`Connection` 객체 하나** — "DB와 이미 연결되어 인증까지 끝난, 바로 SQL을 보낼 수
있는 통신 선"이다. 원격 DB면 ①~④에 **수~수백 ms**가 걸린다.

> ★ 연결의 양 끝은 누구인가 — **WAS(내 애플리케이션) ↔ DB 서버**다. 위 ①~④는 'DB에 접속하는 쪽 = 내
> Spring Boot 앱(WAS)'과 'DB 서버' 사이의 일이다. **웹서버(Nginx/Apache)가 아니다** — 웹서버는 정적 파일
> 담당이고 DB에 직접 연결하지 않는다. DB와 통신하는 건 WAS다(9.2 웹서버 vs WAS).
> ```
> [브라우저] → [웹서버 Nginx] → [WAS = Spring Boot] ──커넥션풀──→ [DB MySQL]
>                                  ↑ 커넥션 풀은 '여기(WAS)'에 있다
> ```

→ **풀이 미리 만들어 두는 것이 바로 이 'Connection 객체(열린 통신 선) N개'다.** 앱 시작 시 ①~④를 미리
N번 수행해 열린 커넥션들을 담아 둔다. 그리고 그 풀은 **WAS(내 애플리케이션) 안에** 있다 — WAS의 JVM
메모리에 `Connection` 객체 N개를 들고 있고, 그에 짝이 되는 세션 N개가 DB 서버 RAM에 생긴다(아래).

#### ③ 'DB 세션 생성'이란 — 디스크가 아니라 'DB 서버 RAM'에 잡는 공간
위 ③의 "세션 생성"이 헷갈리기 쉽다. 이건 **MySQL이 설치된 디렉토리(디스크)에 뭘 쓰는 게 아니라, 실행
중인 MySQL 서버 프로세스의 메모리(RAM)** 에 그 연결 전용 공간을 잡는 것이다.
```
[MySQL 서버 = 실행 중인 프로세스 (RAM)]
  ├─ 공용 메모리(버퍼 풀 등 — 모든 연결이 공유)
  ├─ 세션1 전용 메모리  ← 커넥션1 연결 시 할당
  ├─ 세션2 전용 메모리  ← 커넥션2 연결 시 할당
  └─ ...
```
- 세션 전용 메모리에 담기는 것: 그 연결의 트랜잭션 상태(auto-commit 등), 세션 변수(문자셋·타임존·격리 수준),
  쿼리 작업 버퍼(정렬·조인용), 준비된 statement·임시 테이블 등. "이 연결이 지금 뭘 하는지"의 상태·작업 공간이다.
- **양쪽에 메모리가 잡힌다**: DB 서버(MySQL) RAM의 세션 + 내 앱(JVM) RAM의 `Connection` 객체·소켓 버퍼. 둘이 짝.
- **데이터(행)는 디스크, 세션은 RAM**: 실제 테이블 데이터는 디스크 데이터 파일(`.ibd` 등)에 영구 저장(commit 시
  fsync — 10.4 지속성). 세션/연결은 RAM이라 연결이 끊기면 사라진다. (식당 비유: 데이터 파일=창고의 재료,
  세션=손님 한 팀을 위한 테이블 세팅. 세팅은 손님이 있는 동안만.)

#### 풀 N개 = MySQL 서버 RAM에 세션 N개가 (앱 떠 있는 동안) 상주
풀이 커넥션 10개를 만든다 = **실제로 MySQL에 10번 연결을 맺는다** = MySQL RAM에 **세션 10개가 생긴다**.
- 이 10개는 앱이 떠 있는 **내내 살아있다**(반납해도 끊지 않고 풀에 보관하므로 MySQL 세션도 유지). 쿼리를
  안 날려도 MySQL엔 그 10개가 Sleep 상태로 떠 있다(`SHOW PROCESSLIST`로 확인 가능).
  (HikariCP 기본은 minimumIdle = maximumPoolSize라 보통 풀 크기만큼 처음부터 유지.)
- 정밀하게 나누면: **세션의 기본 골격(연결·세션 변수 등)은 미리 상주**, **무거운 쿼리용 작업 버퍼(sort/join)는
  쿼리할 때 그때 추가**로 잡혔다 풀린다. 그래서 세션당 최대 메모리 ≈ 기본 + 쿼리 시 버퍼.
- 그래서 **연결 수 = DB가 상주시켜야 할 세션 메모리**다. MySQL 메모리 ≈ 공용 메모리 + 세션당 메모리 × 연결 수.
  WAS 3대 × 풀 10 = MySQL 세션 30개 = 그만큼의 RAM. → 이것이 1-5의 "풀을 무작정 키우면 안 되는" 직접적 이유다.

### 1-2. 로우레벨의 비효율 — 매번 만들면 연결이 쿼리보다 비싸다
매 요청마다 커넥션을 새로 만들면, 정작 쿼리는 수 ms인데 **연결 설정(①~④)이 수십~수백 ms**라 배보다
배꼽이 커진다. 게다가 동시 요청이 몰리면 연결 생성이 폭증해 DB가 휘청인다.

### 1-3. 해결 — Connection Pool로 '재사용'
```
[풀 없음]  요청 → ①②③④ 연결 생성(수십~수백 ms) → 쿼리(수 ms) → 연결 끊기   (매번 반복)
[풀 있음]  요청 → 풀에서 '이미 열린' 커넥션 빌림(즉시) → 쿼리 → 풀에 반납       (①②③④ 생략!)
```
- 앱 시작 시 커넥션을 **N개 미리 생성**해 풀에 담는다.
- 요청이 오면 **빌리고(borrow)**, 다 쓰면 **반납(return)**한다(다음 요청이 또 씀). → 매번 생성 비용을 안 치르니 빠르다.
```java
HikariConfig cfg = new HikariConfig();
cfg.setJdbcUrl(url); cfg.setUsername("sa"); cfg.setMaximumPoolSize(10);
HikariDataSource pool = new HikariDataSource(cfg);
try (Connection c = pool.getConnection()) { ... }   // 빌림 -> try 종료 시 '반납'
```
> ★ 핵심: 풀에서 얻은 Connection의 `close()`는 **연결을 끊는 게 아니라 '풀에 반납'**하는 것이다. TCP 선과
> DB 세션은 계속 살아 있어, 다음 요청이 ①~④ 없이 즉시 다시 빌려 쓴다. (PART 8.6 try-with-resources와 같은
> 모양이지만 의미는 '끊기'가 아니라 '반납'.)
>
> 비유: 전화. 커넥션 생성 = 전화 걸어 상대가 받고 본인 확인까지 끝낸 상태(①~④). 풀 = 이미 통화 연결된
> 전화선 10개를 깔아 둠. 빌림 = 새로 다이얼하지 않고 연결된 전화를 들어 용건만 말함. 반납 = 안 끊고 보류했다 다음 사람이 또 씀.

### 1-4. 풀 크기의 의미 — "동시 접속 10명"이 아니다
커넥션 풀 크기 10은 "접속한 사용자 10명"이 아니라 **"그 순간 실제로 DB와 통신(쿼리) 중일 수 있는 게 최대
10개"** 다(한 WAS 기준).
- 사용자 1000명이 사이트를 보고 있어도, 대부분은 그 찰나에 DB를 안 건드린다(화면 보는 중 등). **DB 쿼리를
  날리는 요청만** 커넥션을 빌리고, 보통 수십 ms 만에 반납한다. 그래서 10개로도 초당 수백~수천 요청을 처리한다.
- **11번째 동시 요청**은 "안 됨"이 아니라 **빈 커넥션이 반납될 때까지 대기**한다. 대기가 너무 길면 HikariCP가
  타임아웃(기본 30초)을 낸다(`Connection is not available`) → 이것이 '커넥션 풀 고갈' 장애다.

비유: 커넥션 = 은행 창구 10개. 손님 1000명이 와 있어도 창구에서 실제 업무 중인 건 최대 10명, 나머지는
둘러보거나 대기. 손님이 금방 처리되면 10개로 하루 수천 명을 처리한다.

### 1-5. 풀 크기 ≠ DB의 max_connections, 그리고 적정 크기
- **풀 크기**(예: 10) = *내 WAS 한 대*가 DB에 들 수 있는 연결 수(내가 정함).
- **max_connections**(예: MySQL 기본 151) = *DB 서버 전체*가 받을 수 있는 연결 수.
- WAS가 3대고 각자 풀 10이면 DB엔 최대 30개 연결 → **풀 크기 × WAS 대수 ≤ DB max_connections**여야 한다.
- **풀은 클수록 좋은 게 아니다**: DB의 동시 처리 능력은 CPU·디스크에 한계가 있어, 커넥션이 너무 많으면
  DB가 컨텍스트 스위칭·락 경합으로 오히려 느려진다(PART 7.1 "스레드 많다고 빠르지 않다"와 같은 맥락).
  HikariCP 권장도 보통 코어 수 기반의 작은 값(수십 개)이다. (창구를 100개로 늘려도 직원=DB CPU이 감당 못 하면 더 느려짐.)

### 1-6. 커넥션은 '행(row)'과 무관하다 — 락과는 별개 층위
커넥션과 락은 서로 다른 층위다. 헷갈리지 말자.
- **커넥션(풀)**: "DB에 들어가는 통로". 어느 행을 건드리든 **DB 작업(트랜잭션) 하나당 통로 하나**를 차지(개수 문제).
- **락(row-level)**: "**같은 행**을 동시에 수정할 때만" 작동(충돌 문제).

| 상황 | 커넥션 | 락 |
|---|---|---|
| 10명이 각자 다른 행 수정 | 10개 사용(각 1개) | 충돌 없음 → 동시 진행 |
| 10명이 같은 행 수정 | 10개 사용(각 1개) | 같은 행 락 → 한 명씩 순차 |

→ 행이 같든 다르든 **커넥션은 동시 작업 수만큼(여기선 10개) 똑같이 차지**한다. 11번째 요청은 (행과 무관하게)
커넥션이 없어서 대기한다(락이 아니라 '통로 부족' = 풀 고갈). 즉 **커넥션 차지 기준 = "DB 작업 중이냐"(행 무관)**,
락 작동 기준 = "같은 행을 동시에 건드리느냐".

### 1-7. 풀과 DB 세션·락, 그리고 동시 수정
- **풀의 Connection N개 = DB 세션 N개.** 한 Connection(세션) 안에서 **동시에 두 트랜잭션은 불가**(트랜잭션은 연결 단위 — 10.4).
  단 "한 커넥션=한 트랜잭션"은 서버 내부 규칙이지 사용자 수 제한이 아니다(사용자 둘이면 커넥션도 둘 빌림).
- **DB Lock**: 같은 행을 수정 중인 트랜잭션이 있으면 다른 세션의 같은 행 수정을 막는다(배타 락 X, 읽기는 공유 락 S).
  자바 synchronized와 닮은꼴이고 **DB에서도 데드락이 생긴다**(PART 7.6과 같은 원리, 무대가 DB).

#### 공유 락(S) vs 배타 락(X) — 읽기냐 쓰기냐로 나뉜다
DB 락은 한 종류가 아니라 **읽기/쓰기에 따라 두 종류**가 있다.
- **공유 락(Shared Lock, S락) = "읽기 잠금"**: 내가 읽는 동안 **남도 같이 읽는 건 허용**, 단 **수정은 금지**.
- **배타 락(Exclusive Lock, X락) = "쓰기 잠금"**: 내가 수정하는 동안 **남은 읽기도 쓰기도 다 금지**(나 혼자 독점). '배타(排他)'='남을 배제한다'.

왜 둘로 나눴나 — **읽기는 데이터를 안 바꾸니 여럿이 동시에 해도 안전**하기 때문이다. 락을 한 종류(무조건
독점)로 만들면 "읽기끼리도 서로 기다리는" 낭비가 생긴다. 그래서 읽기끼리는 같이 허용해 동시성을 높였다.
```
읽기 + 읽기 -> 충돌 X (둘 다 안 바꾸니 동시 OK)
읽기 + 쓰기 -> 충돌 O (바꾸는 중에 읽으면 어중간한 값을 봄)
쓰기 + 쓰기 -> 충돌 O (서로 덮어씀)
```

호환성 표 — "이미 걸린 락이 있을 때 내가 새 락을 잡을 수 있나?":

| 내가 원하는 락 ↓ \ 이미 걸린 락 → | 없음 | 공유 락(S) | 배타 락(X) |
|---|---|---|---|
| 공유 락(S, 읽기) | 가능 | 가능 | 대기 |
| 배타 락(X, 쓰기) | 가능 | 대기 | 대기 |

- **S+S만 호환**: 여러 명이 같은 행을 동시에 읽는 건 다 허용.
- **S가 걸린 행에 X 시도 = 대기**: 누가 읽는 중엔 수정 못 함.
- **X가 걸린 행엔 아무것도 못 함**: 수정 중인 행은 남이 읽지도 수정하지도 못함(독점).

언제 걸리나:
- **배타 락(X)**: `UPDATE`/`DELETE`/`INSERT`, 또는 `SELECT ... FOR UPDATE`(읽지만 곧 수정할 거라 미리 독점).
  → 아래 시나리오에서 "id=10에 배타 락"이 이것이다.
- **공유 락(S)**: 격리 수준에 따라 일반 `SELECT`가 읽는 행에 걸리거나, `SELECT ... LOCK IN SHARE MODE`로 명시한다.

> ★ MySQL InnoDB 기본 격리 수준(REPEATABLE READ)에선 평범한 `SELECT`가 사실 락을 거의 안 건다(MVCC라는
> 스냅샷 방식으로 읽음). S/X 락이 또렷하게 드러나는 건 `UPDATE`(X) / `SELECT ... FOR UPDATE`(X) /
> `LOCK IN SHARE MODE`(S) 같은 명시적 상황이다. 개념부터 잡되, 실제 동작은 DB·격리수준마다 다름을 기억하자.

자바 `synchronized`와 비교: synchronized는 읽든 쓰든 **무조건 독점** = 배타 락 한 종류만 있는 셈이다. DB
락은 거기서 한 발 더 나가 **읽기끼리는 같이 허용(S락)** 해 동시성을 높였다(자바의 `ReadWriteLock`이 같은
아이디어 — PART 7의 그 개념의 DB판). 비유: 화이트보드를 '보는 것'(S, 여럿 OK)과 '쓰는 것'(X, 한 명 독점).

**두 사용자가 '다른 글'을 동시 수정 → 막힘 없음**(다른 행이라 락 충돌 0, 커넥션도 각자 빌림).
**두 사용자가 '같은 글'을 거의 동시 수정 → 순차 처리 + 나중 것이 덮어씀(Last Write Wins)**:
```
1) 내 트랜잭션이 id=10에 배타 락 -> '내글' 수정
2) 친구 트랜잭션은 같은 행이라 락 풀릴 때까지 잠깐 대기
3) 내가 commit -> 락 해제 -> '내글' 확정
4) 친구가 락 얻어 '친구글'로 덮어씀 -> commit -> 최종 '친구글'
```
- ★ 락은 "데이터가 뒤섞여 깨지는 것"은 막지만(무결성), "내 수정이 조용히 사라지는 것"은 못 막는다
  (갱신 유실 lost update — PART 7.4와 같은 개념). 락은 '순서 정렬'이지 '사용자 의도 충돌 감지'가 아니다.
- 막으려면 **낙관적 락(@Version)**: `update ... where id=10 and version=1` → 버전이 이미 바뀌었으면 0건
  수정 → 앱이 "다른 사람이 먼저 수정했습니다" 알림(위키·구글독스 원리). JPA `@Version`(이후 PART).

### 1-8. 커넥션 풀 vs 스레드 풀 — 같은 패턴, 다른 대상
| | 스레드 풀 (PART 7.8) | 커넥션 풀 (여기) |
|---|---|---|
| 담는 자원 | 스레드(실행 흐름) | DB 커넥션(통신 선) |
| 비싼 이유 | 스택 ~1MB + OS 등록 | TCP+인증+세션 생성 |
| 푸는 문제 | "작업을 누가 실행하나" | "DB에 어떻게 연결하나" |
| 크기 의미 | 동시 실행 작업 수 | 동시 DB 작업 수 |

**실무에선 협력한다**: 요청 → 스레드 풀에서 스레드 빌림 → 그 스레드가 커넥션 풀에서 커넥션 빌려 DB 작업.
스레드 풀 200 + 커넥션 풀 10이면, 동시 요청 200개여도 DB 작업은 한 번에 10개 → 나머지는 커넥션 대기(병목).
"스레드 많다고 처리량이 비례해 늘지 않는다"(PART 7.1). 비유: 종업원(스레드) 풀 + 주방 출입증(커넥션) 풀.

### 1-9. HikariCP란 — '커넥션 풀'이라는 개념의 실제 구현체
지금까지 말한 '커넥션 풀'은 **개념**이고, **HikariCP는 그 개념을 실제 코드로 만든 라이브러리(제품 이름)** 다.
- **개념(커넥션 풀)**: 연결을 미리 만들어 빌리고·반납·재사용하는 것(1-1~1-4).
- **구현(제품)**: 그 개념을 실제로 만든 라이브러리들 → **HikariCP**, DBCP2, Tomcat JDBC Pool, c3p0 등.
- HikariCP가 그중 가장 빠르고 가벼워서 사실상 표준 → **Spring Boot 2.0부터 기본 커넥션 풀**(starter에 자동 포함).

이름의 유래: **Hikari(光)는 일본어로 "빛"**, CP = Connection Pool. "빛처럼 빠른 커넥션 풀"이라는 뜻이다.
실제로 바이트코드 최적화·락 최소화로 빌림/반납이 매우 빠르고, 코드가 작아 버그가 적다.

PART 8/10의 사상으로 보면 HikariCP는 **`DataSource`(인터페이스, 10.3)의 구현체 `HikariDataSource`** 다.
우리가 코드에서 `DataSource`에만 의존하면(10.3 DIP), 실제로 주입되는 구현이 HikariCP인 것이다.
```java
HikariConfig cfg = new HikariConfig();
cfg.setJdbcUrl(url); cfg.setMaximumPoolSize(10);   // 풀 크기 10
HikariDataSource pool = new HikariDataSource(cfg); // ← 이게 HikariCP 풀(= DataSource 구현)
try (Connection c = pool.getConnection()) { ... }  // 빌림/반납
```
> ★ 당신 프로젝트에서도 이미 동작 중이다. 11.3~11.5 JPA 테스트 로그에 떴던 `HikariPool-1 - Starting...`,
> `HikariPool-1 - Added connection ...`가 바로 그것 — JPA(Hibernate)도 내부적으로 HikariCP 풀을 통해
> DB에 연결한다. Spring Boot에선 보통 `HikariDataSource`를 직접 만들지 않고 `application.yml`만 설정하면
> Spring이 자동으로 HikariCP를 구성해 준다.

Spring Boot 실무 설정(참고):
```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10      # 풀 크기
      connection-timeout: 30000  # 커넥션 못 빌리면 30초 후 타임아웃(풀 고갈 — 1-4)
      max-lifetime: 1800000      # 커넥션 최대 수명(오래된 건 교체) — 아래 ★ 참고
```

> ★ `max-lifetime`은 왜 있나 — 커넥션이 '닳아서'가 아니라 '좀비 커넥션'을 막기 위해서다.
> 커넥션은 TCP 연결 + DB 세션(1-1)이라 물리적으로 마모되진 않는다. 문제는 **앱은 살아있다고 믿고 풀에
> 들고 있는데, 중간에서 몰래 끊겨 있는** 경우다. 끊는 주체가 앱이 아니라 딴 데다:
> ```
> [WAS 풀] ── 살아있다고 믿는 선 ──→ [DB]
>            그런데 몰래 끊기는 경우:
>            - DB가 idle 오래된 연결을 강제 종료 (MySQL wait_timeout)
>            - 방화벽/로드밸런서가 오래된 TCP 연결을 idle timeout으로 끊음
>            - 네트워크 장비가 유휴 연결 정리
> ```
> 끊긴 줄 모르고 빌려 쿼리를 날리면 그제야 에러(`Connection reset`/`Broken pipe`)가 난다 → 멀쩡하던
> 사이트가 간헐적으로 터지는 장애. `max-lifetime`은 **그 전에, 우리가 먼저 통제된 시점에** 수명이 다 된
> 커넥션을 (놀고 있을 때) 조용히 닫고 새것으로 교체하는 '신선도 관리' 장치다.
> ```
> 커넥션 생성 → max-lifetime 경과 → (사용 중 아닐 때) 폐기 → 새 커넥션으로 풀 채움
> ```
> 실무 규칙: **max-lifetime을 DB/인프라가 idle 연결을 끊는 시간보다 짧게** 잡는다. 그래야 DB·방화벽이
> 먼저 끊어 좀비가 되기 전에 우리가 먼저 갈아줄 수 있다(HikariCP 공식 권장도 "그보다 몇 초~몇 분 짧게").
> 비유: 비상용 손전등 배터리를 '닳아서'가 아니라 '정작 필요할 때 죽어있을까 봐' 미리 교체하는 것.

#### 그 'DB가 끊는 시간' — MySQL wait_timeout이란
위에서 "DB가 idle 오래된 연결을 끊는다"고 했는데, 그게 MySQL의 `wait_timeout`이다.

기준은 **"연결을 만든 지 오래"가 아니라 "쿼리를 안 보낸 채 놀고 있은(idle) 지 오래"** 다. 쿼리를 하나라도
보내면 타이머가 리셋된다.
```
연결 → 쿼리 → (놀기 시작) → 아무것도 안 보냄 → 이 '논 시간'이 wait_timeout 넘으면 → DB가 끊음
                            ↑ 쿼리를 보내면 이 타이머가 0으로 리셋
```
(max-lifetime은 '만든 지', wait_timeout은 '논 지' — 둘 다 좀비를 막지만 자르는 잣대가 다르다.)

MySQL에 기본 설정값으로 들어 있고 조정 가능하다:
```sql
SHOW VARIABLES LIKE 'wait_timeout';   -- 기본 28800초 = 8시간
SET GLOBAL wait_timeout = 600;        -- (임시) 10분
```
```ini
# my.cnf / my.ini (영구)
[mysqld]
wait_timeout = 600
```
(`interactive_timeout`은 콘솔 같은 대화형 세션용으로 별도. 클라우드 RDS·방화벽·로드밸런서가 이보다 더
짧은 자체 타임아웃을 거는 경우가 많아, 실제 끊기는 시간은 인프라 전체를 봐야 한다.)

★ "노는 연결을 끊으면 다시 만들 텐데, 그냥 놔두는 게 효율 아닌가?" — 그 논리는 **"그 연결을 다시 쓴다"는
가정에서만** 맞고, 그게 바로 **커넥션 풀의 논리**다(다시 쓸 걸 아니까 안 끊고 들고 있음). 하지만 wait_timeout은
**DB 서버 입장**이라 셈법이 다르다. DB는 자기에게 붙은 수많은 연결이 **앞으로 쓰일지, 그냥 버려졌는지 구분할
방법이 없다.** idle 연결 상당수는 다시 안 쓰인다 — 앱이 크래시/강제종료로 정리 못 하고 사라지거나,
네트워크가 끊겨 양쪽 다 모르거나, 일회성 클라이언트가 떠난 경우다. 이런 건 영원히 안 돌아오는데, 놔두면
RAM·max_connections 슬롯을 **영구히** 점유한다.

| | 그 연결을 다시 쓸 경우 | 그 연결을 안 쓸 경우(버려짐) |
|---|---|---|
| 놔두면 | 이득 (재생성 비용 0) | 손해 (RAM·슬롯 영원히 낭비) |
| 끊으면 | 손해 (다음에 재생성 비용 한 번) | 이득 (자원 즉시 회수) |

wait_timeout을 길게(8시간) 잡은 건 이 절충이다: "8시간이나 놀았다면 다시 안 쓸 확률이 높다"고 보고 끊어서
**'안 쓸 경우의 큰 손해(영구 낭비)'를 막는 보험**이다. 다시 쓸 경우의 손해(재생성 한 번)는 작고, 안 쓸 경우의
손해(영구 점유)는 크니 끊는 게 합리적이다. 정상 앱 연결은 풀의 keepalive(노는 연결에 주기적 신호로 타이머
리셋)와 max-lifetime(끊기 전에 먼저 교체)이 살려두므로 피해가 없다 — **"내가 계속 쓸 연결"은 풀이 살려두고,
"버려진 연결"만 DB가 청소하는 분업**이다.
비유: 헬스장 사물함. "또 올 사람인데 자물쇠 그냥 두지?"는 다시 온다는 보장이 있을 때만 맞다. 헬스장은 누가
다시 올지 모르니 하루 종일 안 열린 사물함은 비워 새 회원이 쓰게 한다(안 비우면 유령 자물쇠로 꽉 참). 매일
오는 단골(정상 앱)은 매일 열어서 안 잘린다.

비유: '커넥션 풀'이 자동차(개념)라면 HikariCP는 특정 모델(예: 테슬라). DBCP2·c3p0는 다른 모델이고,
Spring Boot는 그중 성능 좋은 HikariCP를 기본 출고 모델로 깔아 준다.

### 1-10. max_connections는 조정 가능 / 풀과의 차이 / 초과 시 무슨 일이
1-5에서 "풀 크기 ≠ max_connections"라고 했는데, 셋을 분명히 정리한다.

#### (1) max_connections는 고정값이 아니라 수정 가능한 DB 설정값
MySQL 기준:
```sql
SET GLOBAL max_connections = 500;   -- (임시) 즉시 적용, 단 DB 재시작하면 원래대로
```
```ini
# my.cnf / my.ini  (영구) — 재시작 후에도 유지
[mysqld]
max_connections = 500
```
(PostgreSQL은 `postgresql.conf` 수정 후 재시작, 클라우드 RDS는 파라미터 그룹에서 변경 — 방법은 달라도 다 조정 가능.)

★ 하지만 "늘릴 수 있다 ≠ 늘리는 게 답이다". 무작정 키우면 안 되는 이유:
- **연결마다 RAM을 먹는다**(1-1 ③): max_connections를 1만으로 키우고 실제로 그만큼 차면 MySQL이 메모리 부족(OOM)으로 죽는다.
- **CPU·디스크는 그대로**(PART 7.1과 같은 함정): 연결만 많아도 DB가 동시에 진짜 처리하는 일은 코어 수·I/O에 묶여 있어, 컨텍스트 스위칭·락 경합만 늘어 오히려 느려진다.
- 그래서 실무 정석은 "연결을 무한정 받기"가 아니라 **풀로 적정 수 제한 + 재사용**(1-3). max_connections는 "그 풀들의 합 + 여유"만큼만 잡는다.
- 늘리는 정당한 경우: **WAS 대수가 늘어** 필요 연결이 기본값(151)을 초과할 때. (단 DB RAM이 그 세션들을 감당하는지 확인.)

#### (2) max_connections ≠ 커넥션 풀 — 위치·주체·목적이 다름
둘 다 '연결 개수'라 헷갈리지만 완전히 다른 것이다.

| | 커넥션 풀 (Connection Pool) | max_connections |
|---|---|---|
| 어디에 있나 | WAS(내 애플리케이션) 안 | DB 서버(MySQL) 안 |
| 누가 정하나 | 내가 (HikariCP 설정) | DBA/DB 설정 |
| 무엇인가 | 연결을 미리 만들어 빌리고·반납·재사용하는 보관함 | DB가 동시에 받아줄 수 있는 연결의 최대 개수(상한선) |
| 목적 | 성능(연결 재사용으로 빠르게) | 보호(DB가 감당 못 할 연결 폭주를 막음) |
| 개수 의미 | "내 앱이 들고 있는 연결 수"(예: 10) | "DB 전체가 허용하는 연결 수"(예: 151) |
| 넘으면? | 풀이 비면 요청이 대기(풀 고갈) | DB가 연결 거부(`Too many connections`) |

관계는 하나로 요약된다: **모든 WAS의 풀 크기 합 ≤ DB의 max_connections.**
```
[WAS#1] 풀 30  ┐
[WAS#2] 풀 30  ├→ DB로 가는 연결 총 90개
[WAS#3] 풀 30  ┘
                ↓
[DB MySQL] max_connections = 151  → 90 ≤ 151 → OK
```

#### (3) WAS 대수 × 풀 크기 > max_connections 이면 — DB가 초과분을 거부
예: WAS 3대 × 풀 60 = 180 필요, 그런데 max_connections = 151.
```
[DB] 151개까지만 받고, 152번째 연결 요청부터 거부:
  ERROR 1040 (HY000): Too many connections
```
→ 이 연결을 받으려던 WAS는 커넥션을 못 만들어 에러가 난다.

★ 핵심은 **"언제 터지냐"** — 풀이 연결을 언제 만드느냐에 따라 시점이 다르다:
- **HikariCP 기본(minimumIdle = maximumPoolSize)**: 시작하자마자 풀 크기만큼 연결 시도 → 먼저 뜬 WAS들이 한도를 채우면 **늦게 뜬 WAS가 부팅 단계에서 풀 초기화 실패**.
- **idle을 적게 잡은 경우**: 평소엔 합이 151 이하라 멀쩡하다가, **트래픽 폭주로 여러 WAS가 동시에 풀을 최대로 늘리는 순간** 초과분이 거부됨 → **하필 가장 바쁜 피크에 갑자기 장애**(가장 위험).

사용자 입장 증상: 일부 요청이 DB 연결 실패로 500 에러, 또는 커넥션을 기다리다 타임아웃(`Connection is not available`).
어떤 요청은 되고 어떤 요청은 안 되는 **간헐적/부분적 장애**라 디버깅이 까다롭다. 평소(개발·테스트)엔 멀쩡하다
운영 피크에 터지는 시한폭탄 — PART 7.4 "어쩌다 되는 코드는 위험"과 같은 맥락이다.

막는 법(둘 중 하나 또는 둘 다):
1. max_connections를 올린다 (단 DB RAM이 그 세션들을 감당하는지 확인 — 1-1 ③).
2. 각 WAS의 풀 크기를 줄인다 (합이 한도 안에 들어오게).
3. 공식: **(WAS 대수 × 풀 크기) + 관리·모니터링용 연결 + 여유 ≤ max_connections.**
```
WAS 3대 × 풀 40 = 120, + 여유 20 = 140 ≤ max_connections 151  → 안전
```
비유(콜센터): 회선 151개인데 거래처들이 직통선을 총 180개 깔려고 하면 152번째부터 "회선이 다 찼습니다"로
거부. 평소 통화량 적을 땐 모르다, 다들 동시에 전화하는 피크 때 일부 거래처가 아예 연결 불가가 된다.

## 3. 자기 점검

- **Q. 풀이 '미리 만들어 두는' 커넥션이란 무엇인가?**
  - 내 답: TCP 연결 + 인증 + DB 세션 생성(①~④)을 마친, 바로 SQL을 보낼 수 있는 'Connection 객체(열린 통신
    선)'다. 그 무거운 생성을 미리 N번 해두고 재사용한다.

- **Q. 커넥션 풀이 빠른 이유와, close()의 의미는?**
  - 내 답: 매 요청마다 치르던 연결 생성 비용(TCP+인증+세션)을 안 치르고 이미 열린 걸 빌려 쓰기 때문. 풀에서
    얻은 close()는 끊기가 아니라 '풀에 반납'이라, 다음 요청이 또 재사용한다. (실측 336→39ms)

- **Q. 커넥션 풀 10이면 'DB 동시 접속 10명'인가?**
  - 내 답: 아니다. "그 순간 동시에 DB 작업 중일 수 있는 게 최대 10개"(한 WAS 기준). 접속자 1000명이어도 DB
    쿼리 중인 요청만 빌리고 금방 반납한다. 11번째는 대기(너무 길면 타임아웃=풀 고갈). 풀 크기는 DB의
    max_connections(DB 전체 한도)와 별개라, WAS 여러 대면 합산이 DB 한도를 넘지 않게 한다.

- **Q. 커넥션은 같은 행을 수정할 때만 차지하나?**
  - 내 답: 아니다. 행과 무관하게 'DB 작업(트랜잭션) 하나당 통로 하나'를 차지한다. 10명이 각자 다른 행을
    수정해도 커넥션 10개를 다 쓴다. 락(같은 행)과 커넥션(통로 개수)은 별개 층위.

- **Q. 공유 락(S)과 배타 락(X)의 차이는?**
  - 내 답: 공유 락은 읽기 잠금 — 여럿이 동시에 읽기는 OK, 수정은 금지. 배타 락은 쓰기 잠금 — 한 트랜잭션이
    독점해 남은 읽기도 쓰기도 못 함. 규칙은 S+S만 호환, 나머지(S+X, X+X)는 대기. 둘로 나눈 이유는
    "읽기끼리는 데이터를 안 바꿔 안전하니 동시에 허용해 성능을 높이려고". `UPDATE`는 X락이라 같은 행 동시
    수정이 순차가 된다. (자바 synchronized는 X락 한 종류만 있는 셈.)

- **Q. 두 사용자가 '같은 글'을 거의 동시에 수정하면?**
  - 내 답: 락으로 순차 처리되어 나중에 commit한 쪽이 덮어쓴다(Last Write Wins). 데이터는 안 깨지지만 먼저 쓴
    수정이 조용히 유실된다(lost update). 막으려면 낙관적 락(@Version)으로 "버전이 바뀌었으면 덮어쓰기 거부".

- **Q. 커넥션 풀과 스레드 풀(7.8)은 같은 건가?**
  - 내 답: '풀 패턴'(미리 만들어 빌리고·반납·재사용)은 같지만 대상·목적이 다르다(커넥션=DB 연결, 스레드=실행
    흐름). 실무에선 협력하며, 스레드 풀이 커도 커넥션 풀이 작으면 거기서 병목이 생긴다.

- **Q. 'DB 세션 생성'은 MySQL 설치 디렉토리(디스크)에 잡는 건가?**
  - 내 답: 아니다. 실행 중인 MySQL 프로세스의 RAM에 그 연결 전용 공간(트랜잭션 상태·세션 변수·작업 버퍼)을
    잡는 것이다. 데이터(행)는 디스크 데이터 파일에 영구 저장되지만, 세션/연결은 RAM(휘발성)이다.

- **Q. 풀 10이면 MySQL은 그 10개 세션 메모리를 미리 잡아두나?**
  - 내 답: 그렇다. 풀 10 = MySQL에 실제 연결 10개 = MySQL RAM에 세션 10개가 앱 떠 있는 동안 상주(Sleep
    상태로 유지). 세션 기본 골격은 미리, 무거운 쿼리용 버퍼는 쿼리할 때 추가로 잡힌다. 그래서 연결 수가
    DB RAM을 직접 좌우한다(풀을 무작정 못 키우는 이유).

- **Q. 커넥션은 '웹서버(Nginx) ↔ DB' 연결인가?**
  - 내 답: 아니다. 'WAS(내 Spring Boot 앱) ↔ DB 서버' 연결이다. 웹서버(Nginx)는 정적 파일 담당이라 DB에
    직접 연결하지 않는다. 커넥션 풀도 WAS 안에 있고, WAS가 시작 시 DB로 미리 N번 연결해 둔다.

- **Q. HikariCP는 무엇인가?**
  - 내 답: '커넥션 풀'이라는 개념을 실제로 구현한 라이브러리(제품)다. Hikari(光=빛)처럼 빠른 Connection
    Pool(CP)이라는 뜻. 빠르고 가벼워 Spring Boot의 기본 커넥션 풀이며, `DataSource`(10.3)의 구현체
    `HikariDataSource`다. JPA 테스트 로그의 `HikariPool-1`이 바로 이것.

- **Q. max-lifetime은 왜 있나? 커넥션이 오래 쓰면 닳나?**
  - 내 답: 닳는 게 아니다. 오래 살아있는 커넥션은 DB(wait_timeout)·방화벽·네트워크 장비가 몰래 끊어
    '좀비 커넥션'이 될 위험이 있다. 끊긴 줄 모르고 빌려 쿼리하면 그제야 에러(간헐적 장애). max-lifetime은
    그 전에 우리가 먼저 통제된 시점에 커넥션을 폐기·교체하는 안전장치다. 실무에선 DB/인프라가 idle 연결을
    끊는 시간보다 짧게 잡는다.

- **Q. wait_timeout은 무엇을 기준으로 연결을 끊나? 어디에 설정돼 있나?**
  - 내 답: '연결 만든 지'가 아니라 '쿼리를 안 보낸 채 idle한 지'를 기준으로 끊는다(쿼리하면 타이머 리셋).
    MySQL 기본 설정값(28800초=8시간)이고 `SET GLOBAL`/my.cnf로 조정 가능, 대화형은 `interactive_timeout`.

- **Q. 노는 연결을 끊으면 다시 만들 텐데, 그냥 놔두는 게 효율 아닌가?**
  - 내 답: "그 연결을 다시 쓴다"는 가정에서만 맞고, 그게 풀의 논리다. DB는 붙은 연결이 다시 쓰일지 버려졌는지
    구분 못 한다. 버려진 연결(크래시·네트워크 단절·일회성 클라이언트)을 놔두면 RAM·슬롯을 영원히 낭비하므로,
    "오래 놀았으면 버려진 걸로 간주해 회수"하는 게 합리적이다. 정상 앱 연결은 풀의 keepalive/max-lifetime이
    살려두니 피해 없음.

- **Q. max_connections는 못 바꾸는 고정값인가? 무작정 늘리면 되나?**
  - 내 답: 바꿀 수 있다(`SET GLOBAL` 임시, my.cnf 영구). 하지만 무작정 늘리면 안 된다 — 연결마다 RAM을
    먹어 메모리 부족 위험, CPU·디스크 한계로 연결만 많아도 오히려 느려진다(PART 7.1). 정석은 풀로 적정 수
    제한 + 재사용. WAS 대수가 늘어 기본값을 넘을 때만 (DB RAM 확인하며) 상향한다.

- **Q. max_connections와 커넥션 풀은 같은 건가?**
  - 내 답: 다르다. 커넥션 풀은 WAS(내 앱) 쪽·성능용·"내가 들고 쓰는 보관함"이고, max_connections는 DB 쪽·
    보호용·"DB가 받아주는 상한선"이다. 관계는 '모든 WAS의 풀 크기 합 ≤ DB의 max_connections'.

- **Q. WAS 대수 × 풀 크기 > max_connections 면 무슨 일이 벌어지나?**
  - 내 답: DB가 한도 초과분 연결을 거부한다(`Too many connections`) → 그 연결을 못 받은 WAS는 부팅 실패
    또는 런타임 에러(요청 500/타임아웃). 터지는 시점은 '모든 WAS가 풀을 가득 채우는 순간'이라 평소엔
    멀쩡하다 피크 트래픽에 갑자기 장애가 난다(가장 위험). 막으려면 max_connections 상향(DB RAM 확인) 또는
    풀 축소로 '(WAS 대수 × 풀 크기) + 여유 ≤ max_connections'를 맞춘다.
